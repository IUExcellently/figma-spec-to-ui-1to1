# figma-spec-to-ui-1to1

这个 skill 用于把 Figma 设计输入、节点信息、人工整理规格文档，转成一套可继续开发的高保真页面设计说明书、开发方案与前端实现，并尽可能逼近 100% 的视觉还原。

它不绑定某一个具体业务页面，适用于任意页面、弹窗、抽屉、表单页、详情页、流程页、配置页，也适用于多画板、多状态、多浮层混杂在同一大节点下的情况。它强调先整理规格，再拆解开发方案，最后输出代码；禁止直接跳过规格整理进入自由发挥式开发。

如需补齐输入材料、统一开发提示或交付前自检，请按需读取：

- `references/figma-input-template.md`：整理 Figma 输入、画板归类、布局意图与人工补充规格
- `references/development-checklist.md`：开发完成后的高保真自检
- `references/prompt-template.md`：给 Codex、Kiro 等工具复用的统一提示词模板

## 目标

此 skill 的核心目标固定如下：

- 目标是高保真、接近 1:1，且尽可能逼近像素级的 UI 与交互还原
- 优先保证视觉结构、层级、尺寸、间距、字体、边框、圆角、背景、交互逻辑一致
- 不要为了代码抽象牺牲视觉还原
- 不要擅自美化设计
- 不要擅自脑补未提供的信息
- 高保真不只是颜色和大概尺寸一致，还包括文本排版、视觉中心、定位参考系、z-index、组件默认样式覆盖、状态边界和适配后的布局意图一致

### 固定优先级

优先级顺序固定为：

1. 视觉还原准确性
2. 布局层级准确性
3. 样式细节准确性
4. 交互逻辑准确性
5. 代码复用性

## 输入识别规则

当用户输入中出现以下任意一类信息时，应触发本 skill：

- Figma 链接
- 节点 ID、节点路径、节点层级说明
- 页面结构描述
- 手工整理的尺寸文档
- 样式说明
- 交互说明
- 响应式说明
- 全局组件规范
- 需要生成页面规格说明
- 需要生成高保真页面代码

若同时存在多份材料，应先做输入分层，而不是直接写代码。典型分层如下：

- 全局规则：设计系统约束、跨端行为、组件基线、全局交互
- 主布局规则：项目级页面壳、Header、Sidebar、内容区、断点逻辑
- 页面级规则：某一页面的模块结构、尺寸、布局、表单、状态、业务流程
- 组件级规则：某一块卡片、弹窗、抽屉、标签、时间轴、列表项、组合输入框等

## 固定工作流

### 第一阶段：先理解输入材料

在任何规格整理或代码输出之前，必须先完成以下识别：

1. 识别哪些是全局规则
2. 识别哪些是主布局规则
3. 识别哪些是页面级规则
4. 识别哪些是组件级规则
5. 识别哪些是视觉信息
6. 识别哪些是交互信息
7. 识别哪些是业务规则
8. 识别哪些信息是缺失的
9. 识别哪些数值是 Figma 明确给出的设计原值
10. 识别哪些宽高、间距、坐标、对齐结果其实是布局关系推导出来的视觉结果
11. 识别哪些宽度是固定值，哪些其实应理解为铺满、剩余空间、自适应、最小宽度或最大宽度限制
12. 识别哪些横向容器属于以下布局角色：
   - 单子元素铺满型
   - 左主体右操作型
   - 左固定右铺满型
   - 左工具位中间居中右工具位
   - 左固定中间主体右操作
   - 三列均衡型
13. 识别哪些容器或元素依赖 `absolute`、`fixed`、`sticky` 等精确定位
14. 识别哪些小数尺寸、小数坐标、小数间距会影响边界、滚动或总高度
15. 识别哪些默认样式来自组件库，需要覆盖才能接近设计稿
16. 若一次性输入多个 node-id，先识别它们的结构树关系，而不是先收集样式
17. 识别这些 node-id 之间是否存在父子关系、兄弟关系、共同父容器、状态归属、浮层归属与独立页面归属
18. 识别根节点、一级模块、二级模块以及滚动容器、固定容器、Auto Layout 容器
19. 识别当前链接对应的 node-id 本身属于：
   - page
   - section
   - frame
   - group
   - instance
   - component
   - 其他容器节点
20. 若当前 node-id 是 `section`、大 `frame` 或页面集合节点，先列出它的直接子节点，而不是直接把根节点当成单一页面
21. 识别这些直接子节点中，哪些是不同断点：
   - 1440
   - 1280
   - 1024
   - 768
   - 390
   - 375
   - 414
   - 其他带尺寸语义的画板
22. 识别这些直接子节点中，哪些是业务状态画板，而不是断点画板
23. 识别这些直接子节点中，哪些是弹窗、抽屉、ActionSheet、拨号弹层、图片预览、下拉层、Popover 等浮层节点
24. 识别这些直接子节点中，哪些只是局部状态稿、对比稿、说明稿、待审批稿、已完成稿，而不是默认态
25. 识别本次开发默认应下钻到哪个子节点，而不是直接对根 section 生成代码
26. 识别当前任务到底是：
   - 只做单一断点
   - 做一套响应式断点
   - 做单一默认态
   - 做默认态 + 若干状态态
   - 做主页面 + 独立浮层
27. 若平台返回的是 section 级稀疏结构或高层 metadata，必须继续下钻到目标子节点后，才允许读取详细设计上下文
28. 识别哪些信息可以直接从 Figma 得到，哪些信息其实必须来自：
   - 用户补充说明
   - 人工整理规格文档
   - 现有前端代码
   - 接口文档
   - 业务规则
29. 识别哪些文案是动态拼接文案、计算结果文案或状态文案，而不是静态展示文案
30. 识别哪些规则属于：
   - 接口级联依赖
   - 表单校验
   - 自动回填
   - 自动重置
   - 显隐联动
   - 默认值生成
   - 列表项命名规则
31. 识别哪些图标、图片、插图、业务资源并非来自 Figma 画布本身，而是应来自项目已有资产或组件库
32. 识别哪些行为无法仅凭 Figma 还原，必须以 `【待确认】` 或 `【推断值】` 标记后再推进
33. 识别项目现有实现基线，包括：
   - 全局 CSS / reset / layout 容器
   - UI 框架及其尺寸档位
   - 已引入的自定义主题包、设计 token、变量覆盖
   - 栅格系统、断点系统、表单行高和间距约定
34. 区分哪些宽度只是当前画板下的视觉结果，哪些宽度才是真正需要固定的设计语义
35. 识别哪些组件应优先采用栅格化、弹性布局、宫格布局、列跨度或铺满剩余空间，而不是直接写死宽度
36. 识别 UI 框架组件的内部结构约束，例如：
   - 图标占位
   - 左右 padding
   - suffix / prefix 间距
   - 箭头、清除按钮、状态图标的占位
   - 默认行高与文本垂直居中方式
37. 优先读取项目中的 `package.json`，判断是否已安装 UI 框架、主题包或设计系统依赖
38. 优先扫描项目下常见的公共组件目录与公共样式目录，例如：
   - `src/components`
   - `src/common`
   - `src/shared`
   - `src/styles`
   - `src/theme`
   - `src/assets/styles`
   - `components`
   - `styles`
   - `packages/ui`
39. 判断当前 Figma 元素是否可以明显映射为：
   - 已安装 UI 框架中的现成组件
   - 本地项目已有可复用组件
   - 本地项目已有公共样式能力
   - 上述都不满足的手写模块

若目标是生成接近 `create.md` 的可开发规格文档，或声称“可以直接按 Figma 做接近 100% 的 UI 开发”，则在进入第二阶段前，必须先产出一份“关键真值表”。关键真值表至少应包含：

- 当前链接命中的根节点 ID、名称、节点类型
- 实际实施目标节点 ID、名称、节点类型
- 仅供参考的 sibling 节点 / 组件稿 / 状态稿 / 浮层稿
- 每个纳入开发范围断点的画板尺寸
- 各断点 page-header 的直接可见静态文案
- 各断点筛选区的真实行数、每行元素数量与关键宽度
- 各断点搜索框位置与宽度
- 各断点列表区起始坐标、列表高度、首张卡片尺寸与普通卡片尺寸
- 独立浮层节点的尺寸、挂载归属、触发关系与遮罩关系

在关键真值表之外，还必须同步产出“冲突清单”。冲突清单至少应覆盖：

- Figma 真值与人工整理文档之间的冲突
- 不同断点之间的真实结构差异
- 人工修正与 Figma 旧稿之间的冲突
- 设计真值与工程实现建议之间的冲突

在关键真值表与冲突清单完成前，不允许声称“规格已完整”，也不允许声称“已经可以直接按 Figma 做 100% 还原开发”。

理解输入时必须显式区分以下信息来源：

- 来自用户当前任务中明确补充的最新文字说明
- 来自人工整理文档的直接信息
- 来自 Figma 节点或截图的直接信息
- 来自项目现有全局规范的约束信息
- 来自合理推断的补全信息

若当前平台具备 Figma 读取能力，应优先读取 Figma 节点、尺寸、层级、样式与资源信息；若平台不具备或读取不完整，则以用户提供的人工整理文档为准。任何缺失信息都不能直接伪装成设计原值。

若用户提供的是一个较大的 Figma 节点，且其下包含多个 frame、group、section 或多张视觉稿，必须先执行“画板归类”和“状态识别”，再进入页面模块树整理。禁止默认把这些子画板直接视为同一页面里的并列静态模块。

若用户直接提供的是带 `node-id` 的 Figma 链接，也不能默认认为该 `node-id` 就是“最终要实现的单一页面节点”。必须先判断它是页面 frame、section、页面集合容器、单独状态稿、局部组件还是浮层节点。若当前 `node-id` 下同时存在多个断点、多个业务状态和多个浮层，必须先缩小本次开发目标节点，再继续规格整理与代码输出。

若平台返回“当前选择的是 section 节点，因此仅返回稀疏 metadata，需要继续对其子节点调用 design context”之类的提示，必须把这类提示视为强约束，而不是可忽略备注。此时禁止直接根据 section 根节点输出代码。

对于当前链接命中的 `node-id`，本阶段还必须先完成“node-id 有效性校验”和“实施节点收敛”：

- 必须先判断当前 `node-id` 是否可直接读取到足够支撑开发的 design context、metadata 与 screenshot
- 若当前 `node-id` 只能返回 section 级稀疏结构、空白截图、选择态错误、权限异常或明显缺失的设计上下文，必须显式输出：
  - `node-id` 有效性结果
  - 当前阻塞原因
  - 候选实施节点列表
  - 每个候选节点的证据链
- 不允许在 `node-id` 无效、失焦、返回稀疏结构或无法唯一收敛时，静默回退到旧规格、历史节点、相似 sibling 节点或个人猜测节点
- 一旦本轮范围涉及多个断点、多个状态或多个浮层，必须分别列出每个断点 / 状态 / 浮层的实际实施节点，而不是只写一个笼统的根节点
- 若任务是“基于现有页面调 UI”，则在进入第二阶段前，还必须补充：
  - 当前前端实现入口
  - 需要对照的断点列表
  - 需要对照的状态 / 浮层列表
  - 是否已准备实现后截图对照与差异回写

若用户只提供 Figma 链接和 node-id，而没有补充业务规则、接口依赖、校验逻辑、动态文案生成规则、资产来源和项目组件约束，则默认不能声称已经拿到了“接近 100% 可开发”的完整规格。此时必须显式输出：

- 哪些内容可直接从 Figma 读取
- 哪些内容只能从业务或代码中补齐
- 哪些内容当前只能标记为 `【待确认】`
- 哪些内容只能以 `【推断值】` 继续推进

### 第二阶段：先输出结构化 UI 设计说明书

在写代码前，优先整理一份结构化 UI 设计说明书。说明书至少必须包含以下部分：

1. 文档理解结论
2. 信息来源与优先级说明
3. 关键真值表
4. 冲突清单
5. 根节点 / 实施节点 / 参考节点归类
6. 全局规则
7. 页面整体结构
8. 页面模块树
9. 各模块详细规格
10. 交互与状态规则
11. 表单与校验规则
12. 数据来源与接口依赖
13. 文案与计算规则
14. 显隐、联动、自动回填与自动重置规则
15. 资源引用清单
16. 响应式差异
17. Arco 默认样式覆盖清单
18. 精确定位与 z-index 风险点
19. 视觉归属 / DOM 挂载归属 / 定位参考归属说明
20. 风险点
21. 待确认项
22. 推断值标记
23. 可直接开发性结论
24. node-id 有效性结果
25. 逐断点 / 逐状态 / 逐浮层实施节点清单
26. 项目实现基线（含 `package.json`、公共组件目录、公共样式目录等证据来源）
27. 宽度适配策略
28. 框架组件映射 / `size` / 必要间距覆盖策略

若检测到大节点下存在多张图、多状态稿或浮层稿，说明书还必须新增以下部分，并放在“页面模块树”之前：

15. 画板归类结果
16. 默认态说明
17. 状态清单
18. 浮层归属关系
19. 触发链路说明
20. 哪些状态纳入本次开发范围
21. 哪些状态仅作参考、不直接开发

若检测到用户一次性提供多个 node-id，说明书还必须在“页面整体结构”之前新增以下部分：

16. node-id 归类结果
17. 页面结构树
18. 模块归属关系
19. 父子层级关系
20. 容器布局关系
21. 滚动区 / 固定区划分
22. 浮层归属关系（若存在）

若当前任务是“调现有页面 UI”或“根据既有代码继续逼近 Figma”，说明书还必须新增以下部分：

23. 当前实现与 Figma 的首轮差异清单
24. 需要先修正的高优先级视觉偏差
25. 实现后需要回写到规格文档的差异项

说明书输出要求：

- 先总结输入材料的角色、冲突点与优先级，再写页面规格
- 若存在多 node-id 输入，必须先输出“node-id 归类结果”和“页面结构树”，再输出“页面模块树”
- 若存在多画板或多状态，必须先输出“画板归类结果”，再输出“页面模块树”
- 若用户希望最终拿到类似 `create.md` 的可开发文档，说明书必须写成“可直接落盘为 Markdown 文档”的形式，而不是只给零散摘要
- 对于无法明确映射到 UI 框架现成组件、本地可复用组件或公共样式能力的手写模块，尺寸、坐标、间距、字体、颜色、边框、圆角、背景、布局方向、定位关系不能省略
- 所有推断值统一使用 `【推断值】` 标记
- 所有实现偏离设计原值的地方都要写成“实现偏差”，不能默默吞掉
- 若存在视觉位置突出但依赖父容器定位的元素，必须单独说明其视觉归属、DOM 挂载归属和定位参考归属
- 若某项无法确认但会影响还原度，必须放入“待确认项”
- 若某项不影响结构但需要先推进开发，可保留 `【推断值】` 后继续
- 若某项属于业务逻辑、接口联动、正则校验、自动计算、文案拼装或本地资产引用，必须显式写出其来源，不允许伪装成 Figma 直接信息
- 说明书阶段必须先记录 Figma 真值，再允许写实现判断；不允许先给“更合理的实现方案”，再反过来把它包装成设计事实
- 若项目已引入全局 CSS、UI 框架、自定义主题包或统一尺寸系统，说明书必须先新增“项目实现基线”小节，再决定宽度和间距策略
- 写详细文档前，必须先读取 `package.json`，并扫描常见的公共组件目录与样式目录，再判断当前元素属于哪种实现来源
- 除非用户明确要求“严格按 Figma 组件宽度实现”，否则常规容器、表单控件、按钮、标签、Tabs、筛选项等宽度，不要默认写死为 Figma 截图里的可见宽度
- 图标、图片、插图、徽标、头像等资源尺寸可更严格参考 Figma；但常规组件的外层宽度与内层 padding、icon 间距、箭头占位必须拆开判断
- 若组件来自 UI 框架，必须显式说明该组件采用的是项目尺寸档位、主题 token 覆盖，还是局部样式修正；不允许只根据外层宽度反推内部 padding / margin
- 优先级固定如下：
  1. 若 `package.json` 中已安装 UI 框架，且当前元素可明显映射为框架组件，并且通过 `size` 即可控制主尺寸，则说明书默认不要展开该组件内部的 `padding` / `margin`
  2. 若没有命中 UI 框架，但本地项目下已存在明显可复用组件或公共样式能力，也默认不要展开其内部 `padding` / `margin`
  3. 只有当以上两种都不满足时，才回到“间距不能省略”的细粒度规格
- 若可以明显判断当前元素已被本地项目 UI 框架、自定义主题组件、本地可复用组件或公共样式能力覆盖，说明书中默认不要展开该组件内部的 `padding` / `margin` 数值，优先写：
  - 组件映射关系
  - `size`
  - 背景色
  - 文字颜色
  - `font-size`
  - `font-weight`
  - 换行方式
  - 对齐方式
  - 阴影
  - 边框 / 圆角 / 状态色等必要视觉覆盖
- 只有当框架默认样式与设计稿存在明确差异，且必须局部覆盖时，才额外写该组件的内部间距偏差
- 除非用户明确授权“可以修改公共组件、公共样式或全局 CSS 文件”，否则说明书和开发方案都必须默认以“页面内适配 / 局部覆盖 / 页面级实现”为前提，不能主动把需求落到公共组件或全局样式改造上
- 若某个断点在 Figma 中已经明确给出真实结构，不允许在说明书阶段把它改写成“更灵活的响应式方案”
- 所有直接可见的静态文案必须逐字校对；若出现人工修正、旧稿冲突或动态文案怀疑，必须进入冲突清单或待确认项
- 必须显式区分根节点、实施节点、参考节点，避免把集合根节点与真正要开发的页面节点混为一谈
- 说明书结尾必须输出“可直接开发性结论”，只允许使用以下结论：`可直接开发` / `可开发但无法承诺 100%` / `当前不建议直接开发`

建议说明书的写法遵循以下层次：

- 先写信息来源与页面壳
- 再写画板归类与状态边界
- 再写页面整体结构与模块树
- 再逐块下钻到模块内部规格
- 再补充交互、状态、表单、校验、接口依赖、动态文案、自动计算、资产来源、响应式、默认样式覆盖、定位与层级风险
- 最后列出风险、歧义、冲突和推断

### 第三阶段：再输出开发方案

在结构化 UI 设计说明书之后，再输出开发方案。开发方案至少包括：

- 页面组件树
- 文件结构建议
- 页面拆分策略
- 哪些适合抽组件
- 哪些不适合过度抽象
- 默认态与状态态的实现边界
- 浮层与触发点的归属关系
- Arco 默认样式覆盖清单
- 精确定位与 z-index 风险点
- 固定尺寸与弹性尺寸边界说明
- 项目实现基线说明
- 栅格化 / 弹性 / 宫格布局适配策略
- 常规组件宽度适配策略
- UI 框架 / 本地可复用组件映射与 `size` 优先策略
- 只有在无法映射到现有组件时才需要展开的间距细节
- 允许滚动的容器说明
- 小数尺寸处理策略说明
- 视觉归属 / DOM 挂载归属 / 定位参考归属说明
- 哪些规则依赖业务代码或接口实现，而不是纯静态 UI
- 哪些文案、时间、标题、标签、序号需要动态计算或动态拼接
- 哪些图标、图片或资源需要复用本地资产路径、组件库图标或 Figma 资源

开发方案的判断标准如下：

- 若多个模块仅数据不同，但视觉结构、间距、边框、状态、交互一致，可抽组件
- 若多个模块虽然主题相似，但层级、边距、标题结构、显隐逻辑、交互差异明显，不要为了复用强行合并
- 若抽象会改变 DOM 层级、布局方向、间距控制方式、视觉依赖关系，则不要抽
- 对于 1:1 还原要求高的页面，允许保留更贴近设计结构的页面内局部实现
- 若某些画板仅表示 hover、active、disabled、empty、loading 等状态，不应抽成独立页面组件，应作为状态分支纳入同一页面或同一组件实现
- 若某些画板表示 modal、dialog、drawer、popup、popover、tooltip、dropdown 等浮层，应明确其触发来源与挂载归属，不得当作页面固定区块实现
- 若某个元素视觉上像“页面底部操作栏”或“顶部悬浮区”，但 DOM 上只是父容器内部子节点，必须先还原其挂载归属和定位参考归属，再决定使用 `absolute`、`fixed` 或 `sticky`
- 若某条规则本质上属于“业务实现要求”而不是“设计值”，必须在开发方案中单独标注，避免后续误以为可以从 Figma 自动推导

### 第四阶段：最后再生成代码

代码生成时必须遵守以下规则：

- 以视觉还原优先
- 若项目已引入全局 CSS、UI 框架、自定义主题包或统一尺寸系统，生成代码前必须先读取并遵守这些基线
- 输出代码前，必须先明确哪些地方使用了设计原值，哪些地方使用了 `【推断值】`
- 缺失值必须标记为 `【推断值】`
- 除非用户明确要求“严格按 Figma 组件宽度实现”，否则常规组件宽度默认优先采用页面栅格、父容器宽度、列跨度、`flex` / `grid` / `minmax` 等布局结果
- 图标、图片、插图、装饰资源可以更严格贴近 Figma 原始尺寸；但不要把组件外层宽度直接等价为组件内部 icon 的 padding、margin 或对齐策略
- 若元素已明显命中项目现有 UI 框架组件，优先复用对应组件并用 `size`、颜色、背景、字重、字号等能力逼近设计，不要默认手写内部 `padding` / `margin`
- 除非用户明确授权“可以修改公共组件、公共样式或全局 CSS 文件”，否则不要主动改动这些公共文件；优先通过页面内布局、页面级样式、局部覆盖和现有组件组合完成适配
- 不允许把 Figma 中明确给出的数值“统一优化”
- 不允许把 2 元素、3 元素横向容器在未经角色识别时直接做成等分布局
- 对于导航栏、标题栏、用户信息行等典型结构，必须先判断布局意图，再选择实现方式
- 若使用 Arco Design 组件，必须列出哪些默认样式被覆盖
- 若存在 `absolute` / `fixed` / `sticky` 元素，必须说明参考容器和层级关系
- 若存在滚动，必须明确滚动容器，而不是让页面意外滚动
- 不允许仅因视觉上位于底部、顶部、右下角、右上角，就直接决定使用 `fixed`
- 若元素实际挂载在父容器内部，并依赖父容器边界、宽度、滚动区或 padding 计算位置，应优先按父容器内部定位处理
- 不允许为了复用破坏结构
- 不允许私自改变层级和布局方向
- 不允许擅自减少 DOM 层级
- 不允许擅自合并视觉模块
- 不允许擅自美化
- 不允许擅自补充未提供设计
- 不允许用“工程优化”为理由牺牲还原度
- 在生成代码之前，必须先明确默认态、交互触发态、浮层归属关系，以及哪些状态在本次开发范围内
- 不允许把 hover / active / disabled / empty / loading 等状态图误实现为新的独立页面
- 不允许把弹窗、抽屉、popup 直接实现为主页面固定结构

若用户只要求规格说明，则交付到第二阶段或第三阶段即可；若用户要求直接开发页面代码，也必须先在响应中补出第二阶段与第三阶段，再进入第四阶段。

## 结合输入文档提炼出的关键通用规则

### 1. 主布局与页面壳

- 页面壳与页面内容要分层处理，先明确项目级主布局，再写页面内部结构
- 项目整体优先按 PC 端开发，再做移动端适配
- 主布局优先明确左右结构、上下结构、固定区、自适应区、滚动区
- Sidebar、Header、Content 的尺寸、背景、边框、折叠逻辑、断点逻辑必须单独说明
- 页面内容区的内边距应由页面自己控制，不应盲目依赖全局默认内边距
- 若当前页面继承本项目既有主布局，可优先沿用以下已知基线：Sidebar 宽 220px、左右与顶部区域高度 100vh、Header 高 48px、主内容背景 `rgba(242, 243, 245, 1)`、常规分隔线颜色 `rgba(229, 230, 235, 1)`

### 2. 布局方式判断

- 左右布局：适用于主内容区与辅助信息区、表单区与导航区、信息区与说明区并列场景
- 上下布局：适用于卡片标题与内容、模块堆叠、按钮栏与表单区、标题与说明文案
- 弹性布局：适用于需要两端对齐、内容自适应、按钮从右向左排列、头像与文本并列等场景
- 宫格布局：适用于表单项双列排布、卡片列表、日期项分组等场景
- 移动端下应优先把双列改为单列，右侧辅助区可隐藏，但不能改变主体模块顺序

### 3. 表单与信息展示规则

- 未特别声明时，输入框、选择器等表单控件最小宽度默认按 220px 处理
- 纯信息展示优先使用 `label + value` 横向排布，label 不强行写死宽度，内容左对齐
- 可编辑表单优先使用 `form-item` 上下排布，label 在上，控件在下
- PC 端常见为每行两个表单项，移动端改为每行一个
- 级联依赖选择必须保留先后顺序约束，不能擅自放宽交互
- 必填、正则校验、依赖校验、切换后重置逻辑，必须与业务规则一起整理
- 组合输入框场景下，左侧下拉宽度应尽可能紧凑，但“下拉宽度 + 右侧输入框宽度”必须与普通输入框总宽保持一致；已知参考值可用作优先基线：手机号区号下拉约 70px、证件类型下拉约 90px

### 4. Arco Design 使用与覆盖原则

- Arco Design 作为基础组件库使用，优先复用其基础能力与交互语义
- 当设计稿与 Arco 默认样式不一致时，应局部覆盖样式，而不是改写设计
- 输入框、选择器、多选、日期选择器、组合输入框等，边框、hover、focus 样式应按项目规范统一覆盖
- Button、Tag、Radio、Checkbox、Switch、Modal、Drawer、Table、Tabs、Collapse、Dropdown、Popover 等结构组件也要单独检查默认高度、padding、line-height、圆角、阴影、头尾间距与浮层偏移
- 不要因为使用第三方组件库，就保留与设计稿不一致的默认边框、间距、字号、状态色或层级表现
- 当前项目已知输入类控件覆盖基线可优先沿用：默认边框 `rgba(201, 205, 212, 1)`，hover 保持同色，focus 使用项目绿色 `rgb(0, 170, 166)`

### 5. 全局组件与跨端行为

- 全局 Message 类提示在 PC 端可直接复用 Arco Message
- 移动端应保留与桌面端一致的语义与类型区分，但按移动端视觉与触摸体验调整位置、宽度和遮罩处理
- 跨端行为差异要写成明确规则，而不是运行时临场决定
- 响应式判断应优先复用项目统一的设备类型判断，而不是页面内自行重复定义
- 当前项目已知移动端 Message 基线可优先沿用：消息主体位于页面下方约 35% 位置、最大宽度 88vw、背景 `rgba(0, 0, 0, 0.68)`、图标固定 16x16、图标与内容间距 16px；若传入 `mask`，需阻止触摸穿透

### 6. 常见页面结构拆分方式

- 卡片：标题栏与内容区拆开描述，标题栏包含 icon、标题、右侧操作，内容区单独写 padding 与布局
- 标题与分割线：需要明确出现位置、间距、是否满宽、与上下模块关系
- 操作栏：需明确是否固定在页面底部、按钮顺序、对齐方式、按钮间距
- 抽屉与弹窗：需明确方向、宽度、内容结构、移动端差异、回显逻辑
- 时间轴、步骤导航、右侧指引：需要明确激活态、随填写位置联动规则、移动端显隐策略
- 本项目常见卡片可优先按“标题栏 + 内容区”两层结构描述：标题栏常见高度 40px，内容区常见内边距 20px；若页面存在底部操作栏，按钮通常从右向左排列

### 7. 组件抽象边界

- 只有“结构一致 + 视觉一致 + 状态一致 + 交互一致”的重复块才适合抽象
- 不要为了提炼公共组件而牺牲 DOM 层级、视觉边界、局部布局差异
- 页面中的特殊卡片、渐变块、引导块、组合输入块，哪怕与普通卡片相似，也应先按独立模块处理
- 能在页面内稳定实现 1:1 的情况下，优先保证还原，不急于抽成高度通用组件

### 8. 大节点下的多画板 / 多状态 / 浮层识别规则

当用户提供的 Figma 节点是一个较大的 node-id，且其下包含多个 frame、group、section 或多个视觉稿时，不允许默认把它们视为同一个页面中的并列静态模块，必须先进行“画板归类”和“状态识别”。

#### 8.1 必须先判断以下几类关系

对于大节点下的每一个子画板或子节点，优先判断它属于以下哪一类：

1. 主页面默认态
2. 同一页面的不同状态
   - 空态
   - 加载态
   - 禁用态
   - 选中态
   - hover 态
   - 展开态
   - 收起态
   - 编辑态
   - 提交后态
3. 同一页面触发出来的浮层
   - modal
   - dialog
   - drawer
   - popup
   - popover
   - tooltip
   - dropdown
4. 某个局部组件的不同状态示意
5. 独立页面
6. 仅用于设计说明的对比稿 / 备选稿 / 注释稿

#### 8.2 识别原则

识别时优先参考以下信号：

- 子画板名称中是否包含：
  - default / hover / active / focus / disabled / empty / loading / selected / expand / collapse / open / close / modal / dialog / popup / drawer
- 子画板之间是否只有局部差异，而整体布局基本一致
- 某个子画板是否明显覆盖在另一个页面之上
- 是否存在遮罩层
- 是否存在明显的触发结果关系
- 是否只变化局部组件而非整个页面
- 是否共享相同页面骨架，只变更局部内容或状态

#### 8.3 强制执行规则

- 若多个画板共享同一页面骨架，但局部内容不同，应优先判断为“同一页面的不同状态”
- 若某个画板存在遮罩、浮层、悬浮面板，应优先判断其为“浮层态”，而不是主页面的静态组成部分
- 对每个状态都必须说明它影响的是页面整体、模块局部还是组件局部
- 对每个浮层都必须说明它挂载在哪个触发点上，而不是只写“有弹窗”或“有 popup”
- 对每个状态都必须说明它是否属于首轮开发范围
- 对每个状态都必须说明它是否应作为默认态渲染
- 若无法确认某个子画板是独立页面还是状态稿，必须在“待确认项”中显式指出，不允许私自合并
- 在输出“页面模块树”之前，必须先输出“画板归类结果”
- 若一个大 node-id 中同时包含默认态、弹窗态、popup 展开态、hover 态、disabled 态、loading 态、empty 态，必须先输出状态清单和挂载关系，禁止直接合并成一个静态页面
- 若默认态与状态态共用同一骨架，开发时应优先抽象为“同一页面 / 同一组件的状态分支”，而不是多个独立页面文件
- 在输出代码之前，必须先明确：
  - 默认态是什么
  - 有哪些交互触发态
  - 每个浮层归属于哪个页面或哪个触发点
  - 哪些状态需要开发，哪些只作为说明参考

#### 8.4 输出要求

当检测到大节点下存在多张图或多种状态时，结构化说明书中必须新增以下部分：

1. 画板归类结果
2. 默认态说明
3. 状态清单
4. 浮层归属关系
5. 触发链路说明
6. 哪些状态纳入本次开发范围
7. 哪些状态仅作参考、不直接开发

#### 8.5 禁止行为

- 不允许把弹窗、抽屉、popup 直接当作主页面固定区块
- 不允许把 hover / active / disabled / empty / loading 等状态图误当作新的独立页面
- 不允许把多个状态稿强行拼成一个页面结构
- 不允许忽略遮罩层、触发关系与显隐条件

#### 8.6 section / 页面集合根节点的特殊规则

当用户提供的链接直接命中的是一个 `section`、页面集合根节点或较大的页面容器时，必须额外执行以下识别：

1. 先列出该根节点的直接子节点
2. 判断这些直接子节点分别属于：
   - 断点画板
   - 业务状态画板
   - 浮层节点
   - 局部组件状态稿
   - 独立页面
   - 仅供说明的参考稿
3. 先确定“本次默认实现目标”是哪个子节点
4. 仅对已确认的目标子节点继续获取详细设计上下文

必须重点检查以下信号：

- 根节点名称是否更像一个集合名、模块名、功能名，而不是具体页面名
- 直接子节点名称是否出现明显的断点命名：
  - `1440`
  - `1024`
  - `768`
  - `375`
  - `390`
  - `414`
  - `desktop`
  - `tablet`
  - `mobile`
- 直接子节点名称是否出现明显的业务状态命名：
  - `待我审批`
  - `待处理`
  - `已完成`
  - `已拒绝`
  - `待支付`
  - `空态`
  - `loading`
  - `disabled`
- 是否存在名称明显像浮层的子节点：
  - `弹窗`
  - `drawer`
  - `popup`
  - `ActionSheet`
  - `拨号`
  - `preview`
  - `dialog`

强制规则如下：

- section 根节点默认只用于“归类”和“缩小范围”，不直接用于最终代码实现
- 当 section 下同时存在多个断点画板时，必须先明确本次是做单一断点还是整套响应式
- 当 section 下同时存在默认态和业务状态态时，必须先明确哪一个是默认交付态
- 当 section 下同时存在主页面和浮层节点时，必须拆开说明“主页面节点”和“浮层节点”，不得混写成一页静态结构
- 当 section 下同时存在 PC 与移动端稿时，必须先判断它们是否共享同一页面语义，再决定是否放入同一开发任务

#### 8.7 断点画板、业务状态画板与浮层混合稿识别规则

当同一根节点下同时存在多个断点、多个业务状态和多个浮层时，必须按以下顺序判断：

1. 哪些是同一页面在不同屏宽下的断点版本
2. 哪些是同一端内的业务状态版本
3. 哪些是被页面触发出的浮层
4. 哪些仅作局部说明，不进入本轮开发

识别优先级建议如下：

1. 先按“平台 / 断点”分组
2. 再在每个断点组内按“默认态 / 业务状态态”分组
3. 最后再从每个组中剥离浮层与局部状态稿

必须显式区分以下几类常见情况：

- `1440 / 1024 / 768 / 375`
  - 默认优先判断为同一页面的不同断点
- `375 / 375-待我审批`
  - 默认优先判断为“同一移动端断点下的不同业务状态”，而不是两个独立断点
- `pc端拨号弹窗 / 拨号`
  - 默认优先判断为“不同端各自的浮层实现”，而不是主页面模块
- 一个断点下的多个 `Container`
  - 默认优先判断为同一页面中的列表项或卡片项，不要仅因重复就拆成多个页面级模块

输出时必须额外补充：

1. 断点分组结果
2. 每个断点对应的默认态
3. 每个断点下包含的业务状态
4. 每个断点下包含的浮层
5. 本轮开发覆盖哪些端、哪些状态、哪些浮层

禁止行为如下：

- 不允许把断点画板当作不同页面
- 不允许把业务状态画板当作不同断点
- 不允许把浮层节点直接拼回主页面静态结构
- 不允许在未确认“做哪个端”的前提下，一口气输出整套代码

#### 8.8 Figma 信息与业务信息边界规则

当目标是生成类似 `create.md` 这种“可直接指导开发”的详细设计文档时，必须显式区分：

1. Figma 直接可读信息
   - 尺寸
   - 坐标
   - 间距
   - 布局方向
   - 颜色
   - 字体排版参数
   - 边框
   - 圆角
   - 显式存在的文案
2. 用户或人工文档补充的信息
   - 页面业务目的
   - 字段含义
   - 接口来源
   - 校验规则
   - 级联逻辑
   - 自动回填 / 自动重置规则
   - 动态文案生成公式
   - 本地资产路径
   - 项目已有组件约束
3. 当前无法确认、只能保留为 `【待确认】` 或 `【推断值】` 的信息

必须强调：

- Figma 本身通常不能完整提供接口依赖、正则校验、动态公式、Toast 触发条件、角色权限和本地资源路径
- 若这些规则没有来自用户、文档或现有代码的补充，就不能假装它们是从 node-id 自动读出来的
- 目标是“1:1 接近设计稿”，不是“把所有业务逻辑脑补完整”

当文档中出现以下类型内容时，必须补充“信息来源标记”：

- 级联选择顺序
- 正则校验规则
- 自动生成文案
- 时间有效期计算规则
- 主访客 / 同行人之类的业务角色切换规则
- 列表标题命名规则，如“访客1 / 访客2”
- 图片或图标的本地资产路径
- 设备判断逻辑，如 `mobileType`

### 9. 数值还原与尺寸基准规则

- Figma 中明确给出的宽度、高度、坐标、padding、margin、gap、border-radius、border-width 等值，默认都应被视为高优先级设计原值
- 但当项目已引入全局 CSS、UI 框架、自定义主题包、栅格系统或统一断点容器时，必须先判断这些宽度和高度是“必须固定的设计语义”，还是“当前画板下的布局结果”
- 不允许因为“看起来差不多”就自行把 `13` 改成 `12`、把 `23` 改成 `24`，或把 `10 / 12 / 14` 统一成某个 spacing token
- 必须明确区分两类信息：
  1. 必须严格固定的设计原值
  2. 由布局关系推导出来的视觉结果
- 对于输入框、Select、Button、Tabs、表单项、筛选栏、工具条、卡片容器等常规业务组件，默认先判断其宽度是否应通过页面栅格、父容器宽度、列跨度、`flex`、`grid`、`minmax` 或铺满剩余空间实现
- 除非用户明确要求“严格按 Figma 组件宽度实现”，或 Figma 已经表达了固定宽度语义，否则不要把常规组件的外层宽度直接写死
- 图标、图片、插图、徽标、头像、装饰块等资源，允许更严格参考 Figma 尺寸；但其外围容器仍要先判断是否受项目布局基线约束
- 若使用 UI 框架组件，不允许只根据外层总宽度反推其内部 icon padding、文本左右留白、箭头占位或 suffix / prefix 间距，必须结合组件尺寸档位、主题 token 与局部样式覆盖综合判断
- 若某个尺寸是 Figma Auto Layout 的 padding 或 gap，应优先映射为真实容器内边距和真实间距，不要随意换成绝对定位、额外占位块或其他“视觉上差不多”的实现方式
- 对于 `1px` 分割线、描边、边框等精细元素，不允许为了实现省事而用模糊阴影、半透明背景或近似方案替代
- 对于坐标值，要先判断它是固定画板下的绝对位置，还是某种相对布局结果；但无论如何，Figma 原值本身都必须被记录为高优先级参考
- 若 Figma 中存在小数尺寸：
  - 默认先保留原值
  - 若运行环境或组件限制必须处理，再采用最小影响策略
  - 所有无法 100% 直接映射的地方，都要明确标记为 `【推断值】` 或“实现偏差”
- 不允许把设计原值和布局推导结果混为一谈，更不允许在未说明的情况下自行“规整化”

### 10. 文本与排版还原规则

- Figma 中明确给出的 `font-size`、`font-weight`、`line-height`、`letter-spacing`、`text-align` 都必须优先按原值实现
- 100% 还原不只是颜色和字号一致，还包括排版与视觉中心一致
- 不允许只关注字号和颜色，必须同时关注：
  - 文本是否单行
  - 文本是否允许换行
  - 是否省略
  - 行数限制
  - 文本容器高度
  - 文本在容器中的垂直对齐方式
- 对于按钮、标签、输入框、标题栏、胶囊块中的文字，需要特别检查“视觉居中”，不能只依赖浏览器默认 `line-height`
- 数字、时间、金额、状态文案等容易产生视觉偏移的内容，要单独检查其视觉中心
- 若组件默认样式导致文字看起来比设计稿“略高、略低、略挤、略松”，必须局部修正，而不是接受默认效果
- 若某段文本在设计稿里通过最大宽度、换行、截断形成特定视觉节奏，开发时必须显式还原，而不是让文本无限拉伸
- `font-family` 的落地策略见第 18 节；此处关注的是排版结果而非字体家族名称本身
- 若文案内容本身是动态拼接结果、计算结果或状态文案，必须在说明书中拆成：
  - 文案模板
  - 动态变量来源
  - 计算规则
  - 示例展示
  不能只写最终示例文案

### 11. 定位、层级与参考系规则

- 当设计稿中的元素依赖 `absolute`、`fixed`、`sticky` 等定位时，必须先明确参考容器
- 不允许把依赖定位关系的结构随意改写成 `flex` 或 `grid`
- 对以下元素必须单独检查：
  - 角标
  - 徽标
  - 装饰块
  - 提示层
  - 覆盖层
  - popup
  - dropdown
  - tooltip
  - popover
  - dialog
  - drawer
- 必须明确：
  - 定位参考容器是谁
  - `top / right / bottom / left`
  - `z-index` 层级
  - 是否有遮罩
  - 遮罩覆盖范围
  - 是否阻断底层交互
- 浮层类组件的对齐锚点关系必须被保留，不能只做到“能弹出来”
- 若元素建立了新的定位上下文或堆叠上下文，也必须显式记录，因为这本身就是设计结构的一部分

### 12. 状态开发范围与交付边界

- 在进入代码实现前，必须显式列出：
  - 默认态
  - 本轮必须实现的状态
  - 可以延后实现的状态
  - 仅作为参考、不进入本轮开发的状态
- 不允许把所有状态稿一次性堆叠到页面默认态中
- 如果某些状态需要在开发阶段保留，允许通过 mock 状态开关、注释说明、预览入口等方式管理
- 默认态必须是最终页面的主交付态
- popup、drawer、dialog、hover、disabled、loading、empty 等状态，必须明确其是否属于本轮交付范围
- 若某状态仅用于说明交互关系或未来迭代，不应塞进默认页面结构中

### 13. 信息优先级规则

信息优先级顺序固定如下：

1. 用户当前任务中明确补充的最新文字说明
2. 用户手工整理并明确修正过的规格文档
3. Figma 节点中直接可读的设计值
4. 项目现有全局规范
5. 合理推断值

同时必须遵守以下规则：

- 当多个来源冲突时，必须按这个优先级取值
- 发生冲突时，不允许自己默默选一个值直接实现
- 必须在“风险点 / 待确认项 / 推断值标记”中指出冲突内容
- 若高优先级信息已经修正 Figma 旧稿，则必须以人工修正为准
- 任何低优先级信息都不能在未说明的情况下覆盖高优先级信息
- 当第 1-3 优先级没有明确要求“固定组件宽度”时，第 4 优先级中的项目现有全局规范可以决定常规组件的宽度策略、尺寸档位、内部 padding、图标占位与断点布局方式，但必须在说明书中写明依据

### 14. 禁止工程化外观修正规则

- 本 skill 的首要目标不是“工程结构最优”，而是“设计还原优先”
- 不允许因为下面这些理由就改动视觉结果：
  - 代码更优雅
  - 组件更通用
  - 响应式更方便
  - 样式更统一
  - token 更整洁
  - DOM 层级更少
- 若因为技术实现限制必须偏离设计，必须：
  1. 明确指出偏离原因
  2. 标记为 `【推断值】` 或“实现偏差”
  3. 说明影响范围和对还原度的影响
- 不允许以“工程优化”为理由偷偷改变布局方向、尺寸、对齐方式、默认状态、定位方式或文本排版
- 不允许因为复用方便而把多个视觉块折叠成一个“通用块”
- 但若项目已存在明确的栅格、弹性布局、全局尺寸档位、主题 token 或 UI 框架约束，且高优先级来源没有要求固定组件宽度，则允许在开发方案中采用这些项目基线来实现常规组件宽度与内部留白；前提是必须显式写出“宽度适配策略”和“框架组件映射 / `size` / 必要间距覆盖策略”，不能把工程实现误写成设计原值

### 14.1 Figma 真值抽取与冲突校对规则

- 在规格阶段，AI 的第一责任是抽取并记录 Figma 真值，而不是总结、归纳或优化它
- 关键真值至少包括：节点范围、断点结构、直接可见文案、关键尺寸、关键坐标、列表区与卡片尺寸差异、浮层尺寸与挂载关系
- 若 metadata、截图、人工文档、用户补充说明之间出现冲突，必须先列“冲突清单”，再继续写规格
- 冲突未澄清时，允许继续推进，但只能以 `【推断值】` 或 `【待确认】` 形式存在，不能包装成设计原值
- 若某条规格无法回溯到 Figma 真值、人工修正或用户明确说明，就不能把它写成“页面事实”

### 14.2 逐断点真值优先规则

- 当 Figma 已经提供多个断点画板时，必须逐断点记录真实结构，而不是只写一份抽象版响应式总结
- 断点之间若存在真实差异，例如筛选区换行方式、搜索框位置、列表起始位置、卡片高度差异、提示条显隐差异，必须逐条写出
- 不能因为“响应式实现更统一”就用一个通用布局覆盖多个断点真值
- 只有在 Figma 未明确给出某断点表现时，才允许使用布局意图推导；一旦已有断点真值，断点真值优先于推导
- 任何“统一结构”“自动换行”“改成更灵活布局”的方案，只能进入开发方案中的工程备选，不得覆盖说明书中的设计真值

### 14.3 可见文案逐字校对规则

- Figma 中直接可见的静态文案，必须逐字记录并逐字校对
- 不允许凭页面语义、历史页面经验、相似页面标题或个人理解，把文案近似替换成“差不多”的内容
- 若怀疑某文案是动态文案、拼接文案、状态文案或人工修正文案，必须在说明书中拆出：
  - 文案真值
  - 动态变量来源
  - 是否为动态生成
  - 当前冲突情况
- 若截图、metadata 与人工文档之间的可见文案不一致，必须进入冲突清单，不能直接择一落地

### 14.4 规格阶段禁止主动工程化改稿

- 结构化 UI 设计说明书阶段只负责记录设计真值、结构关系、状态边界和已知业务规则，不负责替设计稿做工程化重构
- 下面这些内容在规格阶段都不能直接写成最终规格：
  - 统一 DOM 结构
  - 改成 `flex-wrap` 更灵活
  - 改成单一响应式方案统一所有断点
  - 为了组件库方便调整真实布局
  - 为了减少分支把真实断点差异抹平
- 如果必须给出工程实现建议，必须明确放在“开发方案”或“实现偏差”中，并注明：
  - 原始 Figma 真值是什么
  - 为什么要偏离
  - 偏离后影响哪些断点、哪些模块、哪些视觉结果
- 规格阶段若出现工程方案覆盖设计真值，视为高风险错误

### 14.5 node-id 有效性与候选节点收敛规则

- `node-id` 只要无法直接支撑“结构树 + 视觉真值 + 截图对照”三件事中的任意一项，就不能直接视为最终实施节点
- 必须显式判断当前 `node-id` 属于以下哪种情况：
  - 可直接实施
  - 仅可读取高层 metadata
  - 只能作为根容器继续下钻
  - 当前失效或无权限
  - 当前截图与命名明显不匹配
- 若当前 `node-id` 不是最终实施节点，必须列出候选节点，并给出每个候选节点的证据链：
  - 命名证据
  - 断点尺寸证据
  - 直接可见文案证据
  - 结构树位置证据
  - screenshot / metadata 匹配证据
- 未完成候选节点收敛前，不允许输出“最终页面模块树”，也不允许进入代码阶段
- 禁止静默沿用旧规格中的历史节点、上一次对话中的节点或肉眼“看起来差不多”的 sibling 节点

### 14.6 实施节点逐断点 / 逐状态验真规则

- 当本轮范围包含多个断点、多个状态或多个浮层时，必须分别记录每个断点 / 状态 / 浮层的实施节点，不能只写一个总节点
- 每个实施节点至少要回收以下证据：
  - design context
  - screenshot
  - metadata
  - 关键可见文案
  - 关键结构真值
- 若某个断点、状态或浮层没有独立实施节点，必须写明它是：
  - 与默认态共用骨架
  - 仅局部状态分支
  - 仅参考稿，不纳入开发
- 若某个断点、状态或浮层仍未取到足够证据，必须降级为 `【待确认】`，不允许继续包装成“已完成验真”

### 14.7 项目实现基线与组件库尺寸适配规则

- AI 在写详细设计文档前，必须先读取项目中的 `package.json`，并扫描常见的公共组件目录、公共样式目录、主题目录与布局目录，再总结实现基线
- 若仓库中已经存在全局 CSS、reset、布局容器、UI 框架、自定义主题包、变量覆盖文件或统一尺寸档位，AI 在写详细设计文档前必须先读取并总结这些基线
- “项目实现基线”至少应覆盖：
  - `package.json` 中识别出的 UI 框架、主题包、设计系统依赖
  - 扫描到的公共组件目录与公共样式目录
  - 全局字体栈和基础字号策略
  - 布局容器宽度、页面边距、栅格列数、断点规则
  - 表单控件、按钮、Tabs、Tag 等常规组件的默认尺寸档位
  - 自定义主题包对圆角、颜色、边框、阴影、padding、行高的覆盖
- 文档里必须显式记录这些基线来自哪些文件或目录，不能只写抽象结论
- 优先级固定如下：
  1. `package.json` 中存在 UI 框架，且当前元素可明显映射为现成组件，并且通过 `size` 即可控制主尺寸
  2. 虽未命中 UI 框架，但项目下存在明显可复用组件或公共样式能力
  3. 上述两者都不满足，只能按手写模块处理
- 对于常规 UI 框架组件，先判断其在当前页面里属于：
  - 固定语义宽度
  - 内容自适应宽度
  - 跟随父容器铺满
  - 栅格列跨度结果
  - 弹性剩余空间结果
- 若某个控件在 Figma 中显示为 `100 x 28`，不能直接得出“代码里必须写死 `width: 100px`”；必须先判断该 `100` 是否只是当前断点下的视觉结果，以及项目已有尺寸系统是否应主导外层宽度
- UI 框架组件内部 icon、箭头、prefix、suffix、clear 按钮、loading 图标、文本留白，不允许仅按外层宽度等比例缩放；应优先遵守项目尺寸档位和主题 token，再做局部覆盖
- 若当前元素可明确映射为 UI 框架现成组件，且通过 `size` 即可控制主尺寸，则详细文档优先输出“组件名 + `size` + 必要视觉覆盖项”，默认不展开内部 `padding` / `margin` 数值
- 若未命中 UI 框架，但当前元素可明确映射为本地可复用组件或公共样式能力，也同样优先输出“组件映射关系 + 视觉覆盖项”，默认不展开内部 `padding` / `margin`
- 只有当既不能映射到 UI 框架组件，也不能映射到本地可复用组件或公共样式能力时，才要求完整展开间距规格
- 图标、图片、插图、业务徽标等资源可以更严格贴近 Figma 尺寸；但输入框、Select、Button、Tabs、表单容器、筛选栏、卡片操作区等常规组件默认优先采用项目布局基线
- 只有当用户明确说明“严格按照设计稿组件宽度实现”，或 Figma 已明确表达固定宽度语义时，才允许把常规组件宽度直接写死到代码中

## 15. 布局意图推断与适配规则

### 15.1 基本原则

- Figma 设计稿通常基于固定画板尺寸（如 `1920×1080`、`1440×900`、`390×844` 等）呈现视觉结果
- 开发时不能只机械照抄当前画板中的绝对宽高
- 必须先判断：
  - 哪些值是固定尺寸
  - 哪些值是布局结果
  - 哪些值是视觉结果
  - 哪些值应该转化为弹性关系
- 适配时优先保持：
  1. 模块层级
  2. 对齐关系
  3. 主要间距关系
  4. 视觉主次关系
  5. 滚动边界关系
- 固定尺寸设计稿中的某些“宽度看起来是定值”的元素，实际可能只是“父容器剩余空间的结果”，必须先判断其角色，再决定是否写死

### 15.2 单子元素横向容器推断规则

- 当一行只有 1 个主要子元素时：
  - 如果它是表单项、内容区、卡片、列表项、输入容器、主信息块，默认铺满父容器可用宽度
  - 如果它是按钮、tag、图标块、状态块、徽标、小型提示块，则不默认铺满，应优先按内容自适应
- 高度优先按设计值实现
- 若设计值存在小数，先判断是否必须取整
- 不能把“所有单元素容器都铺满”写成死规则，要保留组件角色判断

### 15.3 双子元素横向容器推断规则

必须优先按角色关系判断，而不是默认等分：

1. 左主体 + 右操作区
   - 右侧优先内容自适应
   - 左侧占剩余空间
   - 常见于：标题 + 操作按钮、信息 + 编辑按钮、说明文字 + 开关
2. 左固定辅助区 + 右主体
   - 左侧固定或内容自适应
   - 右侧占剩余空间
   - 常见于：头像 + 用户信息、label + 输入框、icon + 文本
3. 左右均为独立小块
   - 两边都内容自适应
   - 仅当设计明确表现为等宽、等分或固定比例时，才做均分

必须强调：

- 双子元素容器不能默认等分
- 若无法确认，应给出候选关系并标记 `【待确认】`

### 15.4 三子元素横向容器推断规则

三子元素布局不能只按数量判断，必须按角色判断：

1. 左工具位 + 中间主体 + 右工具位
   - 常见于：移动端导航栏、标题栏、顶部工具栏
   - 目标优先是“中间内容视觉居中”，不是三等分
   - 必要时允许通过左右占位或绝对定位保证中间居中
2. 左固定信息 + 中间主体内容 + 右操作区
   - 常见于：头像 + 用户信息 + 编辑按钮
   - 左侧固定或内容自适应
   - 中间主体占剩余空间
   - 右侧操作区内容自适应
   - 不应默认实现为三等分
3. 三列并列信息
   - 仅在设计稿明确表达三列均衡时，才做三等分或比例布局
4. 两侧小块 + 中间主体
   - 左右自适应
   - 中间占剩余空间

必须强调：

- 三子元素布局不能只按数量判断
- 必须按角色判断
- 若无法判断，必须给出候选方案并标记 `【待确认】`

### 15.5 小数尺寸、取整与滚动处理规则

- Figma 中存在小数尺寸、小数坐标、小数间距时，不能简单统一向上取整
- 先判断该值是否真的影响边界与滚动
- 处理顺序建议为：
  1. 优先保留原值
  2. 若环境限制必须取整，再做最小影响处理
  3. 若向上取整会导致总高度溢出，则先判断设计是否允许滚动
- 若页面或容器本身就是可滚动设计，则允许溢出并滚动
- 若页面原本不应滚动，则应回查：
  - 是否所有子元素都必须独立向上取整
  - 是否部分间距应保留小数
  - 是否某些高度只是视觉结果而非固定值
- 必须明确：
  - 哪个容器滚动
  - 为什么滚动
  - 滚动方向
  - 固定区与滚动区边界

### 15.6 宽度适配策略

- 能通过 `flex: 1`、`auto`、`min-width`、`max-width`、内容自适应实现的，不要机械写死宽度
- 若项目已有栅格系统、页面容器、断点规则或 UI 框架尺寸档位，优先基于这些项目基线决定常规组件宽度
- 若组件本身存在清晰的 UI 框架 `size` 档位，优先通过 `size` 控制其整体尺寸，而不是拆成手写宽高和内部 `padding`
- 若未命中 UI 框架，但已命中本地可复用组件或公共样式能力，也优先复用既有尺寸语义和现成视觉能力，而不是重新拆内部 spacing
- 主体内容区优先保持连续性和可读性
- 操作区、icon 区、按钮区优先按内容或设计最小值自适应
- 如果某个文本区域在设计稿中不应无限拉长，需增加最大宽度或省略策略
- 要明确“铺满”“自适应”“最大宽度限制”“最小宽度限制”的判断逻辑
- 若某个宽度来自固定画板中的视觉结果，但真实开发中应随父容器伸缩，必须在说明中显式写成“布局推导结果”，而不是伪装成固定设计原值
- 对 UI 框架组件，外层宽度策略与内层 padding / icon 占位 / 文本留白要拆开判断，不要因为 Figma 看到 `100 x 28` 就把内部间距一起机械缩放

### 15.7 常见角色识别

在布局判断前，优先识别每一行内元素属于哪类角色，例如：

- 主体内容
- 工具位
- 操作区
- 状态区
- 固定信息
- 辅助信息
- 装饰区

布局方案必须基于“角色关系”判断，而不是仅基于“元素数量”判断。

## 16. 多 node-id 输入时的结构树解析规则

### 16.1 基本原则

- 一次性输入多个 node-id 时，不允许默认把它们当作平级独立模块直接拼装页面
- 必须先判断这些 node-id 之间是否存在：
  - 父子关系
  - 兄弟关系
  - 同一父容器归属
  - 同一页面不同层级归属
  - 同一组件不同状态归属
  - 主页面与浮层归属
- 页面还原优先依赖“结构树”，而不是“节点样式集合”
- 在多 node-id 场景下，结构树优先级高于单节点样式细节

### 16.2 必须先判断的关系

在读取多个 node-id 时，先判断它们分别属于以下哪类关系：

1. 同一父容器下的兄弟节点
2. 父节点与子节点
3. 同一页面内不同层级节点
4. 同一页面不同状态下的对应节点
5. 主页面节点与浮层节点
6. 独立页面节点
7. 仅供说明的局部组件节点

### 16.3 强制执行顺序

以下顺序必须执行，不允许跳过：

1. 先识别根节点 / 页面根容器
2. 再识别一级模块
3. 再识别二级 / 三级模块
4. 再识别各节点的父子归属
5. 再识别容器布局方式
6. 再识别滚动容器与固定容器
7. 最后才读取子节点样式细节

同时必须明确：

- 不能先读完所有节点样式，再反推结构
- 必须先还原结构树，再还原视觉细节

### 16.4 容器优先原则

- 父容器的布局方式、宽高策略、滚动方式、定位方式，优先级高于子节点细节
- 先看容器是：
  - flex
  - grid
  - absolute 容器
  - fixed / sticky 容器
  - Auto Layout 容器
- 再决定子节点如何排列
- 不允许脱离父容器直接判断子节点的最终布局

### 16.5 多 node-id 下的输出要求

当检测到用户一次性提供多个 node-id 时，在结构化 UI 设计说明书之前，必须先新增以下输出部分：

1. node-id 归类结果
2. 页面结构树
3. 模块归属关系
4. 父子层级关系
5. 容器布局关系
6. 滚动区 / 固定区划分
7. 浮层归属关系（若存在）

### 16.6 禁止行为

- 不允许把本属于同一容器的子节点误当成页面一级模块
- 不允许把子节点当作平级节点直接拼页面
- 不允许忽略 Auto Layout / 父容器布局关系
- 不允许在未确认父子关系时直接输出页面模块树
- 不允许在未确认结构树时直接生成代码

### 16.7 对高保真还原的意义

- 父子层级、容器关系、布局结构，是高保真还原的前置条件
- 若结构树错误，即使尺寸、颜色、边框正确，最终页面也不会真正接近 Figma

### 16.8 section 根节点与直接子层下钻规则

当用户给出的单个 node-id 本身就是 `section`、页面集合节点或大容器节点时，必须把“直接子层下钻”视为结构树解析的一部分，而不是额外可选步骤。

必须执行以下顺序：

1. 先识别当前根节点是不是 section 或页面集合节点
2. 再列出它的直接子节点
3. 再判断这些子节点中哪些是：
   - 断点版本
   - 业务状态版本
   - 浮层
   - 局部组件状态稿
4. 再确定本次目标实现子节点
5. 最后才对目标实现子节点读取详细设计上下文

必须强调：

- section 根节点的高层 metadata 不能替代目标子节点的详细设计上下文
- 若平台明确提示“当前返回是稀疏结构，需要继续对子节点调用 design context”，则必须继续下钻
- 若未完成下钻，不允许直接生成页面规格或代码

## 17. 视觉归属、DOM 挂载归属与定位参考归属识别规则

### 17.1 基本概念区分

必须显式区分以下三种归属，不能混为一谈：

1. 视觉归属
   - 元素看起来属于哪个区域
   - 例如：页面底部操作栏、卡片右上角操作区、弹窗底部按钮区、表单页底部提交区、页面右下角悬浮按钮
2. DOM 挂载归属
   - 元素在结构上实际是谁的子节点
   - 例如：页面根节点的子节点、表单容器的子节点、卡片容器的子节点、弹窗 body 的子节点
3. 定位参考归属
   - 元素的位置是相对于谁计算的
   - 例如：相对页面主容器、相对表单容器、相对卡片容器、相对弹窗容器、相对 viewport

### 17.2 强制判断规则

当某个元素表现为以下任意一种视觉特征时，必须先判断这三类归属关系，再决定实现方式：

- 看起来位于底部、顶部、右下角、右上角
- 悬浮在内容区之上
- 脱离普通文档流
- 压在卡片、表单、弹窗、主内容区上方
- 有遮罩、悬浮、吸底、吸顶、角标、覆盖层特征

### 17.3 实现判断顺序

遇到此类元素时，必须按以下顺序判断：

1. 它在视觉上属于哪个区域
2. 它在结构上实际挂载到哪个父容器
3. 它的位置是相对于哪个容器计算
4. 它是否参与父容器的宽度限制
5. 它是否跟随父容器边界、圆角、滚动区一起变化
6. 它是否应阻断父容器内部交互
7. 最后才决定使用：
   - `absolute`
   - `fixed`
   - `sticky`
   - 普通文档流布局

### 17.4 典型高风险误判

以下情况必须重点防止：

- 视觉上像页面底部栏，就直接实现为 `position: fixed`
- 视觉上贴在容器底部，就误判为全局 footer
- 视觉上悬浮于父容器内，就误判为页面级浮层
- 视觉上是卡片角标，就误判为页面级独立节点
- 视觉上在弹窗底部，就误判为弹窗外部节点

### 17.5 表单页底部操作栏的特殊规则

若某个操作栏视觉上位于页面底部，但实际是挂载在表单页面容器内部，并通过 `absolute` 定位到父容器区域上，则必须按以下方式处理：

- 默认优先判断为“父容器内部的绝对定位子节点”
- 不允许直接实现为 viewport 级 `fixed`
- 必须明确其：
  - 父容器是谁
  - 宽度是否跟随父容器
  - 左右边界是否受父容器 padding 影响
  - 是否受父容器圆角、滚动区域影响
  - 是否覆盖父容器底部内容
  - 是否需要为正文内容预留底部安全空间
- 若它仅在父容器内部吸底，则应优先使用：
  - 父容器 `position: relative`
  - 子节点 `position: absolute`
- 只有当设计明确要求它始终相对 viewport 固定时，才允许使用 `position: fixed`

### 17.6 输出要求

若检测到这类“视觉位置突出但依赖父容器定位”的元素，结构化说明书中必须单独输出：

1. 元素名称
2. 视觉归属
3. DOM 挂载归属
4. 定位参考归属
5. 实现建议
6. 是否影响滚动区
7. 是否需要为正文内容预留空间
8. 是否属于高风险误判点

### 17.7 禁止行为

- 不允许只因元素“看起来在底部”就直接使用 `fixed`
- 不允许忽略其父容器关系
- 不允许把容器内 absolute 元素误做成全局独立模块
- 不允许把视觉归属当作 DOM 归属
- 不允许在未明确定位参考系时直接写代码

## 18. 字体使用与 font-family 忽略规则

- 默认忽略 Figma 设计稿中的 `font-family` 原始值
- Figma 中出现的字体名称，不默认视为必须还原的设计原值
- 不允许 AI 直接把 Figma 中读到的字体名原样写入前端代码
- `font-family` 不作为必须 1:1 还原项
- Figma 中的字体信息仅用于辅助理解视觉风格
- 可参考的信息包括：
  - `font-size`
  - `font-weight`
  - `line-height`
  - `letter-spacing`
  - `text-align`
- 字体家族名称本身只作为参考信息，不直接落地
- 前端实现应优先使用：
  1. 项目中已经明确存在并统一使用的字体方案
  2. 若项目未明确，则使用项目既有全局字体栈
  3. 若仍未明确，则使用系统安全字体栈
- 不允许为了“更像 Figma”而私自引入未知商业字体
- 不允许在未确认授权的情况下把 Figma 字体名写进 CSS
- 若忽略 `font-family` 后，页面视觉出现轻微差异，允许通过以下方式微调：
  - `font-weight`
  - `line-height`
  - `letter-spacing`
  - 容器高度
  - `padding`
  - 文本最大宽度
  - 垂直居中方式
- 应优先通过排版微调去逼近视觉结果，而不是强行绑定字体
- 只有当用户明确说明“该字体已获得授权且项目中已有正式引入”时，才允许把该字体写入代码
- 若用户未明确说明，则统一视为不可直接写死
- 若发现 Figma 中提供了字体名，可以在说明书中注明：
  - `font-family: Figma 原稿中存在，但默认忽略，不直接落地，实际以前端项目字体方案为准`
- 不要把忽略字体这件事悄悄吞掉，要作为规则写明

## 19. 带 node-id 的 Figma 链接输入专项规则

当用户直接提供带 `node-id` 的 Figma 链接时，必须额外遵守以下规则：

### 19.1 不要把“给了 node-id”误解为“范围已经唯一确定”

- `node-id` 只代表当前命中的节点
- 该节点可能是：
  - 单一页面 frame
  - section
  - 页面集合容器
  - 列表模块
  - 卡片模块
  - 弹窗
  - 抽屉
  - 局部组件
- 因此，给了 `node-id` 并不等于已经拿到了最终开发范围

### 19.2 必须先判断当前 node-id 的节点类型

至少先判断以下问题：

1. 当前 node-id 是 page / section / frame / group / instance / component 中的哪类
2. 当前 node-id 是否拥有多个直接子画板
3. 当前 node-id 是否本质上只是一个“素材集合节点”
4. 当前 node-id 是否需要继续下钻到子节点

### 19.3 遇到以下命名模式时的默认判断

- 纯数字尺寸命名：
  - `1440`
  - `1024`
  - `768`
  - `375`
  - `390`
  - `414`
  - 默认优先判断为断点版本
- 尺寸 + 业务语义命名：
  - `375-待我审批`
  - `1024-已完成`
  - 默认优先判断为“特定断点下的业务状态版本”
- 语义化浮层命名：
  - `pc端拨号弹窗`
  - `拨号`
  - `预览`
  - `删除确认`
  - 默认优先判断为浮层节点
- 集合化根节点命名：
  - `列表`
  - `卡片列表`
  - `页面合集`
  - `访客预约`
  - 默认优先判断为可能需要继续下钻的集合节点

### 19.4 平台与端的拆分规则

若一个根节点下同时出现 PC、平板、移动端稿，必须先拆成不同端视角：

- PC 端
- Tablet 端
- Mobile 端

然后在每个端内再判断：

- 默认态
- 业务状态态
- 浮层态
- 局部状态稿

不要直接跨端混合分析。

### 19.5 本轮开发范围确认规则

在进入规格说明前，必须先明确以下至少一项：

1. 本轮只做 PC 默认态
2. 本轮做 PC + Tablet 响应式
3. 本轮做移动端默认态
4. 本轮做某一端默认态 + 指定浮层
5. 本轮做全端，但只覆盖默认态

若用户未明确说明，必须在说明书中主动给出“本轮默认开发范围”的判断，并标记高风险假设。

### 19.6 禁止行为

- 不允许因为链接里已经带了 `node-id`，就跳过范围识别
- 不允许把 section 根节点直接当作最终页面节点
- 不允许把断点稿、业务状态稿、浮层稿混成一个页面结果
- 不允许在未明确默认态和目标断点前直接生成最终代码

### 19.7 进入详细文档前的项目基线检查

- 当任务目标是“生成详细设计文档并继续落代码”时，在 `node-id` 范围确认后，还必须先检查项目中的：
  - 全局样式文件
  - UI 框架
  - 自定义主题包
  - 断点配置
  - 布局容器与栅格系统
- 若项目已存在这些实现基线，详细文档中必须新增：
  - 项目实现基线
  - 宽度适配策略
  - 框架组件映射 / `size` / 必要间距覆盖策略
- 在这些内容未补齐前，不允许把 Figma 画板中看到的常规组件宽度直接写成最终实现宽度

### 19.8 生成 `create.md` 级详细文档时的额外要求

若用户希望 AI 直接生成类似 `create.md` 这种“可直接交给下一轮 AI 或开发者继续落地”的详细设计文档，则必须额外输出以下内容：

1. 关键真值表
2. 冲突清单
3. 根节点 / 实施节点 / 参考节点归类
4. 页面整体描述
5. 按业务模块分组的详细规格
6. 每个模块的布局方式、尺寸、间距、排版、状态与交互
7. 表单项粒度的规则
8. 数据来源与接口依赖
9. 级联逻辑与校验逻辑
10. 自动回填、自动重置、自动命名、自动计算规则
11. 资源引用清单
12. 组件库覆盖要求
13. 响应式差异
14. 可直接开发性结论

此类详细文档必须满足：

- 可直接保存为 Markdown 文件
- 可被下一轮 AI 直接读取并继续生成代码
- 明确区分“设计值”“业务规则”“推断值”“待确认项”
- 文档粒度至少达到模块级和关键字段级，而不是只到页面级摘要

### 19.9 node-id 失效时的处理规则

- 若当前链接中的 `node-id` 无法直接读取、读取结果为空、只返回高层稀疏 metadata、截图与命名明显不匹配，必须立即输出：
  - `node-id` 有效性结果
  - 失效或不足原因
  - 候选节点列表
  - 每个候选节点的证据链
  - 当前是否允许继续推进
- 若尚未收敛到唯一实施节点，不允许：
  - 直接写页面模块树
  - 直接输出最终代码
  - 直接声称“可以按该 node-id 100% 还原”
- 若用户另有人工修正文档，可在文档基础上继续整理规格，但必须显式标注：
  - 当前 Figma 节点读取不足
  - 当前规格中哪些部分来自人工文档而非当前有效 node
  - 当前结果为何不能视为基于 Figma 真值的最终施工图

## 20. 实现后视觉验真与差异回写规则

当任务不仅是“整理规格”，而是“改现有页面 UI”或“输出页面代码”，必须在代码实现后增加一轮视觉验真闭环：

1. 启动本地页面或进入可预览入口
2. 按本轮纳入范围的断点分别查看当前实现
3. 对默认态、首轮实现态和纳入范围的浮层分别截图
4. 将本地截图与 Figma screenshot 逐一对照
5. 输出首轮差异清单，至少覆盖：
   - 结构层级差异
   - 直接可见文案差异
   - 断点布局差异
   - 间距 / 尺寸 / 对齐差异
   - 定位参考系 / 滚动 / 遮罩差异
   - 组件库默认样式残留差异
6. 至少修正一轮高优先级可见差异，再进行二次核对
7. 将本轮已修正项、仍存在差异项和剩余待确认项回写到规格文档

强约束：

- 若没有做过至少一轮“本地实现截图 vs Figma screenshot”对照，不允许声称“已经 100% 还原”
- 若仍存在明显可见差异，不允许把结果包装成“完全一致”
- 若任务依赖已有规格文档，必须把本轮变更、差异和剩余风险回写进该文档，而不是只停留在代码里
- 若当前环境无法完成截图对照，必须明确说明阻塞原因，并降低结论级别为：
  - 可开发但未完成视觉验真
  - 当前不建议直接宣称 100% 还原

## 输出风格要求

- 先结构化说明，再开发
- 不省略关键尺寸和层级
- 不把推断值写成设计原值
- 不擅自增加视觉效果
- 不擅自修改交互逻辑
- 对歧义点、冲突点、实现偏差要明确指出
- 表述尽量清晰、层次化、适合继续喂给 AI 开发

## 建议输出顺序

当用户要求从 Figma 到页面代码的完整流程时，建议输出顺序固定如下：

1. 输入材料理解结论
2. 信息来源与优先级说明
3. node-id 有效性结果
4. 关键真值表
5. 冲突清单
6. 根节点 / 实施节点 / 参考节点归类
7. 逐断点 / 逐状态 / 逐浮层实施节点清单
8. node-id 归类结果（若存在多 node-id 输入）
9. 页面结构树（若存在多 node-id 输入）
10. 模块归属关系（若存在多 node-id 输入）
11. 父子层级关系（若存在多 node-id 输入）
12. 容器布局关系（若存在多 node-id 输入）
13. 滚动区 / 固定区划分（若存在多 node-id 输入）
14. 画板归类结果（若存在多画板 / 多状态 / 浮层）
15. 默认态说明
16. 状态清单
17. 断点分组结果（若存在跨端或多尺寸画板）
18. 本轮开发范围说明
19. 浮层归属关系
20. 触发链路说明
21. 结构化 UI 设计说明书
22. 开发方案
23. Arco 默认样式覆盖清单

27. 代码实现
28. 本地截图与 Figma 对照结果（若本轮实现代码）
29. 差异修正记录与回写结论（若本轮实现代码）
30. 自检结论
31. 风险点、待确认项、推断值清单

在交付代码前，应对照 `references/development-checklist.md` 自检，确保没有出现未标记推断值、布局方向被改写、模块被擅自合并、响应式规则被私改、数值被无说明统一化、文本视觉中心偏移、定位参考系错误或组件库默认视觉残留等问题。
