# Figma 高保真页面开发通用提示词模板

> 用途：可直接复制给 Codex、Kiro 等工具，让 AI 按“先规格、后方案、再代码”的固定流程执行高保真页面开发，并尽可能逼近像素级还原。

---

请基于我提供的 Figma 链接、节点信息、页面结构、尺寸说明、样式说明、交互说明和响应式说明，完成一次高保真页面开发。

默认技术栈固定为：

- Vue 3
- TypeScript
- Arco Design
- `script setup`

目标固定为：

- 高保真、接近 1:1，且尽可能逼近像素级还原 UI 与交互
- 优先保证视觉结构、层级、尺寸、间距、字体、边框、圆角、背景、交互逻辑一致
- 不要为了代码抽象牺牲视觉还原
- 不要擅自美化设计
- 不要擅自脑补未提供的信息
- 高保真不只是颜色和字号一致，还包括文本排版、视觉中心、定位参考系、z-index、组件默认样式覆盖、状态边界和适配后的布局意图一致

优先级顺序固定为：

1. 视觉还原准确性
2. 布局层级准确性
3. 样式细节准确性
4. 交互逻辑准确性
5. 代码复用性

禁止行为：

- 擅自调整布局方向
- 擅自减少 DOM 层级
- 擅自合并视觉模块
- 擅自美化
- 擅自补充未提供设计
- 用“工程优化”为理由牺牲还原度

你必须按以下顺序执行，除非我明确要求停止在某一个阶段，否则不允许跳过：

## 第一阶段：读取并理解输入

先识别并分类以下信息：

- 哪些是全局规则
- 哪些是主布局规则
- 哪些是页面级规则
- 哪些是组件级规则
- 哪些是视觉信息
- 哪些是交互信息
- 哪些是业务规则
- 哪些信息缺失
- 哪些值来自明确设计原值
- 哪些值其实是布局结果或视觉结果
- 哪些宽度是固定值
- 哪些宽度其实是铺满或弹性结果
- 哪些容器属于：
  - 单子元素铺满型
  - 左主体右操作型
  - 左固定右铺满型
  - 左工具位中间居中右工具位
  - 左固定中间主体右操作
  - 三列均衡型
- 哪些容器依赖精确定位
- 哪些小数值可能影响滚动和边界
- 哪些组件库默认样式可能偏离设计稿
- Figma 中是否存在 `font-family` 信息
- 但默认不要把 `font-family` 当作必须还原项
- 必须优先还原字号、字重、行高、字距、对齐方式和文本容器关系
- 除非我明确说明字体已授权且项目中可用，否则不得将 Figma 字体名直接写入代码

当我一次性提供多个 node-id 时，你必须先执行以下动作：

1. 判断这些 node-id 是：
   - 同一页面的不同模块
   - 同一模块的不同层级节点
   - 同一父容器下的兄弟节点
   - 多个状态节点
   - 浮层节点
   - 独立页面节点
2. 输出 node-id 归类结果
3. 输出页面结构树
4. 输出父子层级关系
5. 输出容器布局关系
6. 输出滚动区 / 固定区划分
7. 在确认这些关系之后，才允许继续输出页面模块树和代码

硬约束：

- 如果没有先识别结构树、父子关系和容器布局，不允许直接根据多个 node-id 拼接页面

如果页面中存在以下类型的元素：

- 底部操作栏
- 顶部悬浮区
- 卡片角标
- 右上角操作区
- 覆盖在内容区上的按钮栏
- 吸附在父容器边缘的浮动区

请不要仅根据视觉位置判断实现方式。

你必须先明确：

1. 它的视觉归属
2. 它的 DOM 挂载归属
3. 它的定位参考归属
4. 它是否相对父容器 `absolute`
5. 它是否相对 viewport `fixed`
6. 它是否影响父容器内部滚动
7. 它是否需要正文内容为其预留空间

在这些关系确认之前，不允许直接决定使用 `absolute`、`fixed` 或 `sticky`。

如果信息缺失，请不要自由发挥。所有缺失但又必须补齐的值，都要用 `【推断值】` 标记。

补充要求：

如果我提供的 node-id 对应的是一个较大的 Figma 容器，且其下包含多个 frame / section / 状态稿 / 浮层稿，请先不要直接输出页面模块树，也不要直接写代码。

你必须先完成以下步骤：

1. 识别这个大节点下包含多少张图或多少个子画板
2. 判断它们分别属于：
   - 主页面默认态
   - 同一页面不同状态
   - 浮层态
   - 局部组件状态稿
   - 独立页面
3. 输出“画板归类结果”
4. 输出“默认态 + 状态清单 + 浮层归属关系 + 触发链路”
5. 在归类完成后，才允许继续写页面模块树和开发方案

若无法确认某个画板的归属，必须列入“待确认项”，不得擅自合并处理。

## 多 node-id 结构解析

若一次输入多个 node-id，至少先输出以下内容：

- node-id 归类结果
- 结构树
- 模块归属
- 容器布局关系
- 浮层归属
- 状态归属

## 第二阶段：先产出结构化 UI 设计说明书

在写任何代码之前，先输出一份结构化 UI 设计说明书，至少包含以下内容：

1. 文档理解结论
2. 信息来源与优先级说明
3. 画板归类结果（若存在多画板 / 多状态 / 浮层）
4. 默认态说明
5. 状态清单
6. 浮层归属关系
7. 触发链路说明
8. 全局规则
9. 页面整体结构
10. 页面模块树
11. 各模块详细规格
12. 交互与状态规则
13. 表单与校验规则
14. 响应式差异
15. 风险点
16. 待确认项
17. 推断值标记

若 Figma 中存在字体名，还必须在说明书中单独说明：

- 是否忽略该字体名
- 实际采用哪种字体策略
- 哪些属于可直接还原：
  - 字号
  - 字重
  - 行高
  - 字距
  - 对齐
- 哪些默认忽略：
  - `font-family`

要求：

- 不省略关键尺寸、坐标、间距、层级、字体、颜色、边框、圆角、背景、定位关系
- 不把推断值写成设计原值
- 对歧义点与冲突点要明确指出
- 输出要层次化、可继续直接用于开发

## 第三阶段：再产出开发方案

在结构化 UI 设计说明书之后，再输出开发方案，至少包括：

- 页面组件树
- 文件结构建议
- 页面拆分策略
- 哪些适合抽组件
- 哪些不适合过度抽象
- Arco 默认样式覆盖清单
- 精确定位与 z-index 风险点
- 固定尺寸与弹性尺寸边界说明
- 默认态与状态态的实现边界
- 允许滚动的容器说明
- 小数尺寸处理策略说明
- 视觉归属 / DOM 归属 / 定位参考归属说明

要求：

- 允许保留贴近设计结构的实现
- 不要为了复用牺牲 1:1 还原
- 不要因为组件抽象破坏 DOM 层级、布局方向与局部间距
- 明确哪些数值可以写死，哪些应通过布局关系推导

## 第四阶段：最后生成代码

最后再生成代码，并遵守以下要求：

- 输出代码前，必须先明确哪些地方使用了设计原值，哪些地方使用了 `【推断值】`
- 以视觉还原优先
- 不允许把 Figma 中的字体名称直接硬编码进 CSS
- 若未获得明确授权，不允许新增字体资源
- 默认使用项目字体方案或系统安全字体栈
- 允许通过排版微调逼近视觉，而不是通过绑定未知字体逼近视觉
- 不允许私自改变层级和布局方向
- 不允许私自修改交互逻辑
- 不允许省略设计中明确给出的样式细节
- 对缺失值继续保留 `【推断值】` 标记
- 不允许把 Figma 中明确给出的数值“统一优化”
- 不允许把 2 元素、3 元素横向容器未经角色识别就直接做成等分布局
- 对于导航栏、标题栏、用户信息行等典型结构，必须先判断布局意图，再选择实现方式
- 若使用 Arco Design 组件，必须列出哪些默认样式被覆盖
- 若存在 `absolute` / `fixed` / `sticky` 元素，必须说明参考容器和层级关系
- 若存在滚动，必须明确滚动容器，而不是让页面意外滚动
- 若某个元素视觉上像底部操作栏、顶部悬浮区或局部吸底工具条，必须先判断它是否只是父容器内部的 `absolute` 子节点，而不是直接做成全局 `fixed`
- 优先复用 Arco Design 的基础能力，但样式必须按设计稿覆盖

如果你发现我提供的输入不足以支撑 1:1 还原，请先在“待确认项”中列出，并继续输出已知部分的结构化说明，不要直接跳过，也不要自由发挥。

最后，在代码输出完成前，请根据高保真自检逻辑再做一次核对，确认以下问题没有出现：

- 页面层级被改写
- 区块被合并
- 尺寸和间距偏移
- 表单布局走样

- 存在未标记的 `【推断值】`
