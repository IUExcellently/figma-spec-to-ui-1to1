# Figma 高保真页面开发通用提示词模板

> 用途：可直接复制给 Codex、Kiro 等工具，让 AI 按“先规格、后方案、再代码”的固定流程执行高保真页面开发，并尽可能逼近像素级还原。

---

请基于我提供的 Figma 链接、节点信息、页面结构、尺寸说明、样式说明、交互说明和响应式说明，完成一次高保真页面开发。

默认技术栈固定为：

- Vue 3
- TypeScript
- Arco Design
- `script setup`

目标固定为：

- 高保真、接近 1:1，且尽可能逼近像素级还原 UI 与交互
- 优先保证视觉结构、层级、尺寸、间距、字体、边框、圆角、背景、交互逻辑一致
- 若项目已存在全局 CSS、UI 框架、自定义主题包、断点容器或栅格系统，必须先参考这些项目基线，再决定常规组件的宽度、内部间距和尺寸档位
- 不要为了代码抽象牺牲视觉还原
- 不要擅自美化设计
- 不要擅自脑补未提供的信息
- 高保真不只是颜色和字号一致，还包括文本排版、视觉中心、定位参考系、z-index、组件默认样式覆盖、状态边界和适配后的布局意图一致

优先级顺序固定为：

1. 视觉还原准确性
2. 布局层级准确性
3. 样式细节准确性
4. 交互逻辑准确性
5. 代码复用性

禁止行为：

- 擅自调整布局方向
- 擅自减少 DOM 层级
- 擅自合并视觉模块
- 擅自美化
- 擅自补充未提供设计
- 用“工程优化”为理由牺牲还原度

你必须按以下顺序执行，除非我明确要求停止在某一个阶段，否则不允许跳过：

在进入第二阶段前，必须先完成一轮“关键真值抽取与冲突校对”。

关键真值表至少包含：

- 当前链接命中的根节点 ID / 名称 / 节点类型
- 实际实施目标节点 ID / 名称 / 节点类型
- 仅供参考的 sibling 节点 / 组件稿 / 状态稿 / 浮层稿
- 每个纳入开发范围断点的画板尺寸
- 各断点 page-header 的直接可见静态文案
- 各断点筛选区的真实行数、每行元素数量与关键宽度
- 各断点搜索框位置与宽度
- 各断点列表区起始坐标、列表高度、首张卡片尺寸与普通卡片尺寸
- 浮层节点的尺寸、挂载归属、触发关系与遮罩关系

冲突清单至少包含：

- Figma 真值与人工整理文档的冲突
- 不同断点之间的真实结构差异
- 人工修正与 Figma 旧稿的冲突
- 设计真值与工程实现建议的冲突

若关键真值表和冲突清单还没有产出，不允许声称“规格完整”或“可以直接 100% 还原开发”。

在进入第二阶段前，还必须先完成一次 `node-id` 有效性校验。至少输出：

- `node-id` 有效性结果
- 当前链接命中的根节点
- 实际实施目标节点
- 仅供参考的 sibling 节点 / 状态稿 / 浮层稿 / 组件稿
- 若当前 `node-id` 无法直接实施，候选节点列表与证据链

硬约束：

- 如果当前 `node-id` 只能返回高层稀疏 metadata、空白截图、错误截图、错误选择态或明显不足的设计上下文，不允许静默沿用旧规格、历史节点或相似 sibling 节点
- 如果尚未收敛到唯一实施节点，不允许直接进入页面模块树，更不允许直接写代码

## 第一阶段：读取并理解输入

先识别并分类以下信息：

- 哪些是全局规则
- 哪些是主布局规则
- 哪些是页面级规则
- 哪些是组件级规则
- 哪些是视觉信息
- 哪些是交互信息
- 哪些是业务规则
- 哪些信息缺失
- 哪些值来自明确设计原值
- 哪些值其实是布局结果或视觉结果
- 项目现有实现基线是什么：
  - `package.json` 中安装了哪些 UI 框架、主题包或设计系统依赖
  - 项目下有哪些常见公共组件目录与公共样式目录
  - 全局 CSS / reset
  - UI 框架及尺寸档位
  - 自定义主题包 / token 覆盖
  - 栅格系统 / 断点系统 / 页面容器
- 哪些宽度是固定值
- 哪些宽度其实是铺满或弹性结果
- 哪些常规组件宽度不应直接写死，而应采用栅格、弹性、宫格、列跨度或铺满剩余空间
- 哪些图标、图片、插图可以更严格参考 Figma 尺寸
- 哪些元素可以明显映射为项目现有 UI 框架组件
- 这些框架组件应优先使用哪个 `size`
- 哪些 UI 框架组件的内部 icon、padding、箭头、suffix / prefix 占位需要按组件尺寸档位与主题 token 适配
- 哪些容器属于：
  - 单子元素铺满型
  - 左主体右操作型
  - 左固定右铺满型
  - 左工具位中间居中右工具位
  - 左固定中间主体右操作
  - 三列均衡型
- 哪些容器依赖精确定位
- 哪些小数值可能影响滚动和边界
- 哪些组件库默认样式可能偏离设计稿
- 当前链接命中的根节点类型
- 当前 `node-id` 是否可直接读取到足够的 design context / screenshot / metadata
- 实际实施目标节点
- 仅供参考的 sibling 节点 / 组件稿 / 状态稿 / 浮层稿
- 每个断点 / 状态 / 浮层对应的实际实施节点
- 每个纳入开发范围断点的关键真值
- 所有直接可见静态文案
- Figma 真值与人工文档的冲突项
- Figma 中是否存在 `font-family` 信息
- 但默认不要把 `font-family` 当作必须还原项
- 必须优先还原字号、字重、行高、字距、对齐方式和文本容器关系
- 除非我明确说明字体已授权且项目中可用，否则不得将 Figma 字体名直接写入代码

当我一次性提供多个 node-id 时，你必须先执行以下动作：

1. 判断这些 node-id 是：
   - 同一页面的不同模块
   - 同一模块的不同层级节点
   - 同一父容器下的兄弟节点
   - 多个状态节点
   - 浮层节点
   - 独立页面节点
2. 输出 node-id 归类结果
3. 输出页面结构树
4. 输出父子层级关系
5. 输出容器布局关系
6. 输出滚动区 / 固定区划分
7. 在确认这些关系之后，才允许继续输出页面模块树和代码

硬约束：

- 如果没有先识别结构树、父子关系和容器布局，不允许直接根据多个 node-id 拼接页面
- 如果 Figma 已经明确给出某个断点的真实结构，不允许在规格阶段把它改写成“更灵活”的响应式方案
- 规格阶段只负责记录设计真值；任何工程优化、统一结构、自动换行方案，都只能放到第三阶段的开发方案中，且不得覆盖真值
- 如果当前 `node-id` 无效或尚未收敛到唯一实施节点，不允许继续包装成“已拿到完整 Figma 真值”

如果页面中存在以下类型的元素：

- 底部操作栏
- 顶部悬浮区
- 卡片角标
- 右上角操作区
- 覆盖在内容区上的按钮栏
- 吸附在父容器边缘的浮动区

请不要仅根据视觉位置判断实现方式。

你必须先明确：

1. 它的视觉归属
2. 它的 DOM 挂载归属
3. 它的定位参考归属
4. 它是否相对父容器 `absolute`
5. 它是否相对 viewport `fixed`
6. 它是否影响父容器内部滚动
7. 它是否需要正文内容为其预留空间

在这些关系确认之前，不允许直接决定使用 `absolute`、`fixed` 或 `sticky`。

如果信息缺失，请不要自由发挥。所有缺失但又必须补齐的值，都要用 `【推断值】` 标记。

补充要求：

如果我提供的 node-id 对应的是一个较大的 Figma 容器，且其下包含多个 frame / section / 状态稿 / 浮层稿，请先不要直接输出页面模块树，也不要直接写代码。

你必须先完成以下步骤：

1. 识别这个大节点下包含多少张图或多少个子画板
2. 判断它们分别属于：
   - 主页面默认态
   - 同一页面不同状态
   - 浮层态
   - 局部组件状态稿
   - 独立页面
3. 输出“画板归类结果”
4. 输出“默认态 + 状态清单 + 浮层归属关系 + 触发链路”
5. 在归类完成后，才允许继续写页面模块树和开发方案

若无法确认某个画板的归属，必须列入“待确认项”，不得擅自合并处理。

## 关键真值抽取与冲突校对

在进入第二阶段前，至少先输出以下内容：

- `node-id` 有效性结果
- 关键真值表
- 冲突清单
- 根节点 / 实施节点 / 参考节点归类
- 逐断点 / 逐状态 / 逐浮层实施节点清单
- 逐断点关键真值记录

5. 根节点 / 实施节点 / 参考节点归类
6. `node-id` 有效性结果
7. 逐断点 / 逐状态 / 逐浮层实施节点清单
8. 画板归类结果（若存在多画板 / 多状态 / 浮层）
9. 默认态说明
10. 状态清单
11. 浮层归属关系
12. 触发链路说明
13. 全局规则

25. 待确认项

27. 可直接开发结论

若 Figma 中存在字体名，还必须在说明书中单独说明：

- 是否忽略该字体名
- 实际采用哪种字体策略
- 哪些属于可直接还原：
  - 字号
  - 字重
  - 行高
  - 字距
  - 对齐
- 哪些默认忽略：
  - `font-family`

要求：

- 对于无法明确映射到 UI 框架现成组件、本地可复用组件或公共样式能力的手写模块，不省略关键尺寸、坐标、间距、层级、字体、颜色、边框、圆角、背景、定位关系
- 不把推断值写成设计原值
- 对歧义点与冲突点要明确指出
- 输出要层次化、可继续直接用于开发
- 规格阶段不允许把工程优化方案写成设计事实
- 所有直接可见的静态文案必须逐字校对并记录
- 若存在根节点、实施节点与参考节点，必须分别写明，不能混写
- 若任务是调现有页面 UI，必须先说明“当前实现与 Figma 的首轮差异”，不能跳过现状对照
- 除非我明确要求“严格按 Figma 组件宽度实现”，否则常规容器、按钮、输入框、Select、Tabs、Tag 等宽度不要默认写死为 Figma 当前画板中的可见宽度
- 图标、图片、插图可以更严格贴近 Figma 尺寸；但常规组件的外层宽度与内部 padding、icon 间距、箭头占位必须拆开判断
- 若项目已经引入 UI 框架与主题包，必须先说明采用了哪些尺寸档位、token 覆盖或局部样式修正
- 你必须先读取 `package.json`，并扫描常见的公共组件目录与样式目录，再判断当前元素属于哪种实现来源
- 优先级固定如下：
  1. `package.json` 中已安装 UI 框架，且当前元素可明显映射为框架组件，并且通过 `size` 即可控制主尺寸
  2. 虽未命中 UI 框架，但本地项目下已存在明显可复用组件或公共样式能力
  3. 以上两种都不满足，才按手写模块处理
- 若命中前两种情况，详细文档默认不要展开该组件内部的 `padding` / `margin`，优先写组件映射、`size`、背景色、颜色、字号、字重、换行方式、对齐方式、阴影和必要视觉覆盖
- 只有当以上两种情况都不满足时，才要求把间距作为不可省略项展开记录
- 除非我明确说明“可以修改公共组件、公共样式或全局 CSS 文件”，否则不要主动把实现方案落到这些公共文件上，优先通过页面内适配、局部覆盖和现有组件组合解决

- Arco 默认样式覆盖清单
- 精确定位与 z-index 风险点
- 固定尺寸与弹性尺寸边界说明
- 项目实现基线
- 栅格化 / 弹性 / 宫格布局适配策略
- 常规组件宽度适配策略
- UI 框架 / 本地可复用组件映射与 `size` 优先策略
- 仅在无法映射到现有组件时才展开的间距细节
- 默认态与状态态的实现边界
- 允许滚动的容器说明
- 小数尺寸处理策略说明
- 视觉归属 / DOM 归属 / 定位参考归属说明
- 本地截图与 Figma 对照计划
- 差异修正后如何回写规格文档
- 若工程实现与关键真值存在偏离，必须单独列出偏离点与影响范围

要求：

- 允许保留贴近设计结构的实现
- 不要为了复用牺牲 1:1 还原
- 不要因为组件抽象破坏 DOM 层级、布局方向与局部间距
- 明确哪些数值可以写死，哪些应通过布局关系推导

## 第四阶段：最后生成代码

最后再生成代码，并遵守以下要求：

- 输出代码前，必须先明确哪些地方使用了设计原值，哪些地方使用了 `【推断值】`
- 以视觉还原优先
- 不允许把 Figma 中的字体名称直接硬编码进 CSS
- 若未获得明确授权，不允许新增字体资源
- 默认使用项目字体方案或系统安全字体栈
- 允许通过排版微调逼近视觉，而不是通过绑定未知字体逼近视觉
- 不允许私自改变层级和布局方向
- 不允许私自修改交互逻辑
- 不允许省略设计中明确给出的样式细节
- 对缺失值继续保留 `【推断值】` 标记
- 不允许把 Figma 中明确给出的数值“统一优化”
- 若项目已存在全局 CSS、UI 框架、自定义主题包或断点容器，必须先按项目基线实现常规组件宽度，再用局部覆盖逼近视觉
- 除非我明确要求“严格按 Figma 组件宽度实现”，否则常规组件不要默认写死宽度
- 图标、图片、插图等资源可以更严格参考 Figma 尺寸，但不要把组件外层宽度直接等价为组件内部 icon 的 padding、margin 或对齐策略
- 若元素已明显命中项目现有 UI 框架组件，优先复用对应组件并使用 `size`、颜色、背景、字号、字重等能力逼近设计，不要默认手写内部 `padding` / `margin`
- 若未命中 UI 框架，但已命中本地可复用组件或公共样式能力，也优先复用这些既有能力，不要默认重新手写内部 spacing
- 除非我明确说明“可以修改公共组件、公共样式或全局 CSS 文件”，否则不要主动修改这些公共文件
- 不允许把 2 元素、3 元素横向容器未经角色识别就直接做成等分布局
- 对于导航栏、标题栏、用户信息行等典型结构，必须先判断布局意图，再选择实现方式
- 若使用 Arco Design 组件，必须列出哪些默认样式被覆盖
- 若存在 `absolute` / `fixed` / `sticky` 元素，必须说明参考容器和层级关系
- 若存在滚动，必须明确滚动容器，而不是让页面意外滚动
- 若某个元素视觉上像底部操作栏、顶部悬浮区或局部吸底工具条，必须先判断它是否只是父容器内部的 `absolute` 子节点，而不是直接做成全局 `fixed`
- 优先复用 Arco Design 的基础能力，但样式必须按设计稿覆盖
- 若关键真值仍有未解决冲突，不允许把结果包装成“已完成 100% 还原”
