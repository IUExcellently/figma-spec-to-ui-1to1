# create.md 级详细设计文档输入模板

> 用途：可直接复制给 Codex、Kiro 等工具，强制 AI 先生成一份接近 `create.md` 粒度的详细设计文档，再基于该文档继续生成代码。
>
> 适用场景：
>
> - 你提供的是 Figma 链接 + `node-id`
> - 你希望 AI 先沉淀一份可直接保存为 Markdown 的详细设计说明
> - 你希望下一轮 AI 或开发者拿着这份文档继续实现代码
> - 你希望尽可能提高 1:1 还原度，并降低 AI 误判结构、状态、浮层、业务规则来源的概率
>
> 重要提醒：
>
> - 启用这套流程后，token 消耗会显著增加
> - 尤其在多 node-id、多画板、多断点、多状态稿场景下，耗费会更明显

---

请使用 `figma-spec-to-ui-1to1`。

我的目标不是让你直接写代码，而是：

1. 先基于我提供的 Figma 链接和 `node-id`，输出一份可直接落盘为 Markdown 的详细设计文档
2. 该文档的粒度要接近 `create.md` 这种可开发文档
3. 在详细设计文档完成后，再基于该文档生成代码

你必须遵守以下规则：

## 零、先检查项目实现基线

如果项目已经引入全局 CSS、UI 框架、自定义主题包、断点系统、栅格系统或统一尺寸档位，你必须先读取并总结这些基线，再决定常规组件的宽度、内部间距与尺寸档位。

你至少要先判断：

- 项目是否存在全局 layout 容器
- `package.json` 中安装了哪些 UI 框架、主题包或设计系统依赖
- 项目下有哪些常见公共组件目录和公共样式目录
- 项目是否存在统一的表单控件尺寸档位
- 项目是否存在自定义主题包、token 覆盖或变量文件
- 页面是否应优先采用栅格、弹性、宫格、列跨度或铺满剩余空间
- 哪些元素可以明显映射为项目现有 UI 框架组件
- 这些框架组件应优先使用哪个 `size`
- 哪些资源可以更严格参考 Figma 尺寸：
  - icon / 图片 / 插图 / 徽标 / 头像
- 哪些常规组件不应默认写死宽度：
  - 输入框 / Select / Button / Tabs / Tag / 表单容器 / 筛选栏 / 卡片操作区

## 一、先判断当前 node-id 的真实范围

你不能因为我已经提供了 `node-id`，就默认认为范围已经唯一确定。

你必须先判断当前 `node-id` 是：

- 单一页面 frame
- section
- 页面集合根节点
- 组件节点
- 浮层节点
- 局部状态稿节点

如果当前节点是 `section`、页面集合根节点或大容器，必须先列出其直接子节点，再做归类，不允许直接生成代码。

## 二、先完成范围缩小和归类

如果当前节点下有多个子画板，你必须先区分：

- 哪些是断点版本
- 哪些是默认态
- 哪些是业务状态态
- 哪些是浮层节点
- 哪些是局部状态稿
- 哪些只是参考稿

如果同时存在 PC、平板、移动端稿，必须先按端分组，再在每个端内判断默认态、状态态和浮层态。

在未完成这些归类前，不允许直接进入代码阶段。

## 三、先生成 create.md 级详细设计文档

你必须先输出一份详细设计文档，文档至少包含以下内容：

1. 输入材料理解结论
2. 信息来源与优先级说明
3. `node-id` 归类结果
4. 页面结构树
5. 画板归类结果
6. 断点分组结果
7. 本轮开发范围说明
8. 页面整体描述
9. 页面模块树
10. 各模块详细规格
11. 交互与状态规则
12. 表单与校验规则
13. 数据来源与接口依赖
14. 文案与计算规则
15. 显隐、联动、自动回填与自动重置规则
16. 资源引用清单
17. 项目实现基线（含 `package.json`、公共组件目录、公共样式目录等证据来源）
18. 宽度适配策略
19. 框架组件映射 / `size` / 必要间距覆盖策略
20. 响应式差异
21. Arco 默认样式覆盖清单
22. 精确定位与 z-index 风险点
23. 视觉归属 / DOM 挂载归属 / 定位参考归属说明
24. 风险点
25. 待确认项
26. 推断值清单

## 四、必须严格区分信息来源

在详细设计文档中，你必须明确区分：

- 哪些是 Figma 直接可读信息
- 哪些是我补充的业务规则
- 哪些依赖现有代码、接口或组件库
- 哪些是 `【推断值】`
- 哪些是 `【待确认】`

不允许把以下内容伪装成 Figma 自动读取结果：

- 接口级联关系
- 正则校验
- Toast 触发条件
- 自动回填 / 自动重置规则
- 时间有效期计算规则
- 主次角色切换规则
- 列表标题命名规则
- 本地资产路径
- 设备判断逻辑

## 五、详细文档写法要求

- 文档必须适合直接保存为 Markdown
- 文档粒度至少到模块级和关键字段级
- 不要只写页面级摘要
- 所有推断值必须显式标记为 `【推断值】`
- 无法确认但会影响还原度的内容必须进入 `【待确认】`
- 所有实现偏离设计原值的地方，都必须写成“实现偏差”
- 除非我明确要求“严格按 Figma 组件宽度实现”，否则常规组件宽度不要默认写死为 Figma 当前画板里的可见宽度
- 图标、图片、插图、徽标、头像等资源可以更严格参考 Figma 尺寸；但常规组件的外层宽度与内部 padding、icon 间距、箭头占位必须拆开判断
- 若项目中已有 UI 框架和主题包，文档里必须先写清楚采用了哪些尺寸档位、token 覆盖和局部样式修正策略
- 你必须先读取 `package.json`，并扫描常见的公共组件目录与样式目录，再判断当前元素属于哪种实现来源
- 优先级固定如下：
  1. `package.json` 中已安装 UI 框架，且当前元素可明显映射为框架组件，并且通过 `size` 即可控制主尺寸
  2. 虽未命中 UI 框架，但本地项目下已存在明显可复用组件或公共样式能力
  3. 以上两种都不满足，才按手写模块处理
- 若命中前两种情况，详细文档默认不要展开该组件内部的 `padding` / `margin`，优先写组件映射、`size`、背景色、颜色、字号、字重、换行方式、对齐方式、阴影和必要视觉覆盖
- 只有当以上两种情况都不满足时，才要求把间距作为不可省略项展开记录
- 除非我明确说明“可以修改公共组件、公共样式或全局 CSS 文件”，否则不要主动把方案落到这些公共文件上，优先通过页面内适配、局部覆盖和现有组件组合解决

## 六、完成文档后再生成代码

在详细设计文档输出完之后，再基于该文档生成代码。

代码阶段继续遵守以下要求：

- 高保真、接近 1:1，还原度尽可能接近 100%
- 不擅自调整布局方向
- 不擅自减少 DOM 层级
- 不擅自合并视觉模块
- 不擅自补充未提供设计
- 不因为工程抽象牺牲还原度
- 对 `absolute` / `fixed` / `sticky` 先判断视觉归属、DOM 挂载归属、定位参考归属
- 若使用 Arco Design，必须显式列出哪些默认样式被覆盖
- 默认忽略 Figma 中的 `font-family`，优先还原字号、字重、行高、字距、对齐与容器关系
- 若项目已引入全局 CSS、UI 框架、自定义主题包、断点系统或统一尺寸档位，必须先按项目基线实现常规组件宽度，再通过局部覆盖逼近视觉
- 除非我明确要求“严格按 Figma 组件宽度实现”，否则常规组件不要默认写死宽度
- 不要把组件外层宽度直接等价为组件内部 icon 的 padding、margin 或对齐策略
- 若元素已明显命中项目现有 UI 框架组件，优先复用对应组件并使用 `size`、颜色、背景、字号、字重等能力逼近设计，不要默认手写内部 `padding` / `margin`
- 若未命中 UI 框架，但已命中本地可复用组件或公共样式能力，也优先复用这些既有能力，不要默认重新手写内部 spacing
- 除非我明确说明“可以修改公共组件、公共样式或全局 CSS 文件”，否则不要主动修改这些公共文件

## 七、最终输出顺序

你必须按以下顺序输出：

1. create.md 级详细设计文档
2. 开发方案
3. 代码实现
4. 自检结论
5. 风险点、待确认项、推断值清单
