# 输入分层与范围识别

## 适用时机

当任务处于以下任一阶段时，优先加载本文件：

- 用户刚给出 Figma 链接、node-id、页面结构说明
- 一次性输入多个 node-id
- 一个大节点下包含多个 frame、group、section、状态稿或浮层稿
- 需要判断哪些是设计原值、哪些是人工修正、哪些是推断值

## 先做输入分层，不要直接写代码

至少先区分以下信息：

- 全局规则
- 主布局规则
- 页面级规则
- 组件级规则
- 视觉信息
- 交互信息
- 业务规则
- 缺失信息
- 接口依赖信息
- 动态文案与计算规则
- 资源来源信息

## 多 node-id 场景的强制顺序

如果用户一次性给了多个 node-id，先判断：

1. 是否存在父子关系
2. 是否存在兄弟关系
3. 是否属于同一父容器
4. 是否属于同一页面的不同层级
5. 是否是同一页面的不同状态
6. 是否存在主页面与浮层关系

在这些关系确认前，不要直接拼页面。

## 大节点下多画板 / 多状态的强制顺序

如果一个大节点下包含多张图或多个 frame，先做“画板归类”：

- 主页面默认态
- 同一页面不同状态
- 浮层态
- 局部组件状态稿
- 独立页面
- 仅供参考的对比稿

然后再输出页面模块树。

## 带 node-id 链接的专项判断

如果用户直接提供的是带 `node-id` 的 Figma 链接，必须先判断：

1. 当前 node-id 是单一页面 frame，还是 section / 页面集合根节点
2. 当前节点下是否存在多个直接子画板
3. 这些直接子画板中，哪些是断点版本
4. 哪些是业务状态版本
5. 哪些是浮层节点
6. 本次默认实现目标需要下钻到哪个子节点

如果当前 node-id 只是 section 或素材集合根节点，不要直接输出代码。

## 断点、状态、浮层的常见识别信号

- `1440 / 1024 / 768 / 375 / 390 / 414`
  - 默认优先判断为断点版本
- `375-待我审批 / 1024-已完成`
  - 默认优先判断为特定断点下的业务状态版本
- `弹窗 / ActionSheet / 拨号 / popup / drawer`
  - 默认优先判断为浮层

若同一根节点下同时存在这三类内容，先按“断点 -> 默认态 / 业务状态 -> 浮层”顺序拆分。

## 信息优先级

默认按以下优先级取值：

1. 用户当前任务里的最新说明
2. 用户手工整理并修正过的规格文档
3. Figma 节点或截图中的直接信息
4. 项目既有全局规范
5. `【推断值】`

有冲突时要显式说明，不要默默选一个值实现。

## Figma 信息与业务信息的边界

如果用户希望最终拿到类似 `create.md` 这种详细设计文档，必须额外区分：

- 哪些内容可直接从 Figma 读取
- 哪些内容来自用户补充或人工整理文档
- 哪些内容依赖现有代码或接口
- 哪些内容当前只能作为 `【待确认】` 或 `【推断值】`

特别要警惕以下内容不要伪装成 Figma 自动读取结果：

- 接口级联关系
- 正则校验
- Toast 触发条件
- 动态文案计算公式
- 主次角色切换规则
- 本地资产路径
- 设备判断逻辑

## 输出要求

在进入代码前，至少先输出：

- 输入材料理解结论
- 信息来源与优先级说明
- node-id 归类结果（若存在）
- 页面结构树（若存在）
- 画板归类结果（若存在）
- 断点分组结果（若存在）
- 本轮开发范围说明
- 默认态说明
- 状态清单
- 浮层归属关系
- 数据来源与接口依赖
- 文案与计算规则
- 显隐、联动、自动回填与自动重置规则
- 资源引用清单

## 禁止行为

- 不要把多个 node-id 默认当成平级页面模块
- 不要把弹窗、抽屉、popup 直接当作页面固定结构
- 不要把 hover、disabled、loading、empty 等状态稿当成独立页面
- 不要把缺失信息直接脑补成设计原值
- 不要把 section 根节点直接当作最终页面节点
- 不要把断点稿、业务状态稿、浮层稿混成同一个页面结果
