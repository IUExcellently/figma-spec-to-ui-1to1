# 结构树、布局意图与定位参考系

## 适用时机

当任务已经进入页面结构分析、模块拆解、布局还原阶段时，加载本文件。

## 先还原结构树，再还原样式

优先级固定如下：

0. 项目实现基线
1. 页面根节点 / 根容器
2. 一级模块
3. 二级 / 三级模块
4. 父子层级关系
5. 容器布局方式
6. 滚动区 / 固定区
7. 样式细节

不要先读完所有样式再反推结构。

如果项目已经引入全局 CSS、UI 框架、自定义主题包、栅格系统或断点系统，先总结这些基线，再判断页面里哪些宽度是固定语义，哪些只是当前画板下的布局结果。

## 布局意图判断

不要仅按元素数量决定布局，要先识别角色关系。

常见角色包括：

- 主体内容
- 工具位
- 操作区
- 固定信息
- 辅助信息
- 状态区
- 装饰区

### 双子元素横向容器

优先判断以下几类：

- 左主体右操作
- 左固定右铺满
- 左右都内容自适应

不要默认做成等分。

### 三子元素横向容器

优先判断以下几类：

- 左工具位 + 中间主体 + 右工具位
- 左固定信息 + 中间主体 + 右操作区
- 明确的三列均衡信息

不要因为有三个元素就直接三等分。

## 视觉归属、DOM 挂载归属、定位参考归属

以下三者必须分开判断：

- 视觉归属：它看起来属于哪个区域
- DOM 挂载归属：它实际是谁的子节点
- 定位参考归属：它的位置相对谁计算

遇到以下元素时，必须先判断三类归属再决定实现方式：

- 底部操作栏
- 顶部悬浮区
- 角标
- 右上角操作区
- 覆盖在内容区上的按钮栏
- 吸附在父容器边缘的浮动区

## `absolute` / `fixed` / `sticky` 判断顺序

1. 它视觉上属于哪里
2. 它结构上挂在哪里
3. 它位置相对谁计算
4. 它是否参与父容器宽度限制
5. 它是否跟随父容器滚动和边界变化
6. 最后才决定用哪种定位方式

## 高风险误判

- 视觉上在底部，就直接做成 viewport 级 `fixed`
- 视觉上像 footer，就误判为页面全局 footer
- 视觉上悬浮在卡片上方，就误判为页面级浮层
- 视觉上在弹窗底部，就误判为弹窗外部节点

## 输出要求

这一阶段至少要明确：

- 项目实现基线
- 页面模块树
- 容器布局关系
- 滚动区 / 固定区划分
- 高风险定位点
- 哪些宽度是固定值
- 哪些宽度是布局推导结果
- 哪些常规组件应采用栅格 / 弹性 / 宫格布局
- 哪些 UI 框架组件需要拆开判断外层宽度与内层 padding / icon 占位
