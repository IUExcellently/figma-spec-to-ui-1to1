# 代码实现、自检与交付边界

## 适用时机

当任务已经完成规格说明和开发方案，准备进入代码阶段时，加载本文件。

## 固定交付顺序

进入代码前，先确认以下内容已经存在：

- 结构化 UI 设计说明书
- 开发方案
- 默认态说明
- 状态开发范围
- 风险点 / 待确认项 / 推断值清单

如果这些内容缺失，不要直接写代码。

## 推断值规则

- 缺失但又必须补齐的值，统一标记 `【推断值】`
- 实现偏离设计原值的地方，要显式写成“实现偏差”
- 不要把 Figma 中明确给出的数值统一优化成 token

## 字体规则

- 默认忽略 Figma 中的 `font-family`
- 不要在未确认授权的情况下把字体名直接写进代码
- 优先通过以下方式逼近视觉：
  - `font-size`
  - `font-weight`
  - `line-height`
  - `letter-spacing`
  - 文本容器高度
  - `padding`
  - 垂直居中方式

## Arco Design 覆盖规则

如果使用 Arco Design，至少检查以下组件的默认样式是否偏离设计稿：

- `Input`
- `Select`
- `DatePicker`
- `Textarea`
- `Button`
- `Tag`
- `Modal`
- `Drawer`
- `Tabs`
- `Table`

要显式列出哪些默认样式被覆盖，不要把默认视觉残留当成正常结果。

## 状态开发边界

进入代码前必须明确：

- 默认态
- 本轮必须实现的状态
- 可以延后实现的状态
- 仅作说明参考的状态

不要把所有状态稿一次性堆进默认态页面。

## 交付前自检

交付前至少检查以下问题：

- 是否改写了页面层级
- 是否擅自合并了视觉模块
- 是否误改布局方向
- 是否遗漏推断值标记
- 是否错误使用 `fixed`
- 是否保留了组件库默认样式残留
- 是否错误把字体名直接写进代码
- 是否明确了滚动容器

若需要更细的检查项，进一步参考 `references/development-checklist.md`。
