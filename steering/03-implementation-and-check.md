# 代码实现、自检与交付边界

## 适用时机

当任务已经完成规格说明和开发方案，准备进入代码阶段时，加载本文件。

## 固定交付顺序

进入代码前，先确认以下内容已经存在：

- 结构化 UI 设计说明书
- 开发方案
- 默认态说明
- 状态开发范围
- 风险点 / 待确认项 / 推断值清单

如果这些内容缺失，不要直接写代码。

## 推断值规则

- 缺失但又必须补齐的值，统一标记 `【推断值】`
- 实现偏离设计原值的地方，要显式写成“实现偏差”
- 不要把 Figma 中明确给出的数值统一优化成 token
- 但若项目已存在全局 CSS、UI 框架、自定义主题包、尺寸档位或断点容器，要先遵守这些项目基线，再决定常规组件宽度与内部间距
- 进入代码前，先检查 `package.json` 与常见公共组件目录、公共样式目录，确认当前元素是命中 UI 框架组件、本地可复用组件，还是必须手写

## 字体规则

- 默认忽略 Figma 中的 `font-family`
- 不要在未确认授权的情况下把字体名直接写进代码
- 优先通过以下方式逼近视觉：
  - `font-size`
  - `font-weight`
  - `line-height`
  - `letter-spacing`
  - 文本容器高度
  - `padding`
  - 垂直居中方式

## Arco Design 覆盖规则

如果使用 Arco Design，至少检查以下组件的默认样式是否偏离设计稿：

- `Input`
- `Select`
- `DatePicker`
- `Textarea`
- `Button`
- `Tag`
- `Modal`
- `Drawer`
- `Tabs`
- `Table`

要显式列出哪些默认样式被覆盖，不要把默认视觉残留当成正常结果。

同时还要额外检查：

- 组件外层宽度是否被误写死为 Figma 截图中的可见宽度
- 组件内部 icon、箭头、prefix / suffix、左右 padding 是否被机械按外层宽度缩放
- 是否本应采用 `flex` / `grid` / 栅格列跨度，却被误写成固定宽度
- 是否忽略了项目自定义主题包对圆角、边框、行高、spacing 的覆盖
- 对可明确映射到 UI 框架现成组件或本地可复用组件的元素，是否优先使用了组件映射和 `size`，而不是在详细文档里展开内部 `padding` / `margin`
- 在未获用户明确授权时，是否避免主动修改公共组件、公共样式和全局 CSS 文件

## 状态开发边界

进入代码前必须明确：

- 默认态
- 本轮必须实现的状态
- 可以延后实现的状态
- 仅作说明参考的状态

不要把所有状态稿一次性堆进默认态页面。

## 交付前自检

交付前至少检查以下问题：

- 是否改写了页面层级
- 是否擅自合并了视觉模块
- 是否误改布局方向
- 是否遗漏推断值标记
- 是否错误使用 `fixed`
- 是否保留了组件库默认样式残留
- 是否错误把字体名直接写进代码
- 是否明确了滚动容器
- 是否先参考了项目实现基线
- 是否把常规组件宽度误写死
- 是否把组件内部间距误当成外层宽度的线性缩放结果

若需要更细的检查项，进一步参考 `references/development-checklist.md`。
