# Onboarding

## Step 1: 判断任务是否适用

当任务满足以下任一条件时，启用本 power：

- 用户提供 Figma 链接、node-id、节点路径、截图观察结果
- 用户要求高保真、1:1、像素级、接近设计稿地还原页面
- 用户希望先输出规格说明、再输出开发方案、最后再生成代码
- 用户明确提到多画板、多状态稿、弹窗态、浮层态、多 node-id 混合输入

如果任务只是普通的低保真原型、创意发挥式页面设计，或不需要遵循设计稿约束，则不要使用本 power。

## Step 2: 明确本 power 的定位

本 power 是文档与流程约束型 power：

- 不提供 MCP 服务器
- 不要求额外 CLI 安装
- 不负责自动读取 Figma 节点
- 核心作用是约束 AI 的分析顺序、输出结构和高保真实现边界

## Step 3: 固定执行顺序

启用本 power 后，默认按以下顺序工作：

1. 先理解输入材料
2. 再输出结构化 UI 设计说明书
3. 再输出开发方案
4. 最后再生成代码
5. 交付前按高保真自检逻辑复核

除非用户明确要求只停留在前面的某个阶段，否则不要跳过前置阶段直接写代码。

## Step 4: 遇到带 node-id 的 Figma 链接时的额外判断

如果用户提供的是带 `node-id` 的 Figma 链接，必须额外判断：

1. 当前 `node-id` 是单一页面 frame，还是 section / 页面集合根节点
2. 当前节点下是否混有多个断点版本
3. 当前节点下是否混有业务状态版本
4. 当前节点下是否混有浮层节点
5. 本次默认实现目标应该下钻到哪个子节点

如果当前节点只是一个 section 或素材集合根节点，先做归类，不要直接输出代码。

## Step 5: 生成详细设计文档时的边界

如果目标是生成类似“可直接给下一轮 AI 或开发者继续落地”的详细 Markdown 设计文档，必须显式区分：

- Figma 直接可读信息
- 用户或人工文档补充的信息
- 来自现有代码或接口约束的信息
- 当前只能标记为 `【待确认】` 或 `【推断值】` 的信息

如果项目已经引入全局 CSS、UI 框架、自定义主题包、布局容器或断点规则，还必须先补一段“项目实现基线”：

- `package.json` 中识别出的 UI 框架、主题包、设计系统依赖
- 扫描到的公共组件目录与公共样式目录
- 全局样式 / reset / 容器规则
- UI 框架尺寸档位
- 自定义主题包覆盖项
- 栅格系统与断点系统
- 常规组件宽度与内部间距的实现边界

不要把业务逻辑、接口依赖、正则校验、动态公式、本地资产路径伪装成从 Figma 自动读取的结果。

# When to Load Steering Files

- 整理 Figma 输入、识别多 node-id / 多画板 / 多状态、处理信息优先级时，加载 `steering/01-input-and-scope.md`
- 还原页面结构树、布局角色、滚动区、定位参考系时，加载 `steering/02-structure-and-layout.md`
- 进入代码阶段、处理 Arco 默认样式覆盖、字体策略和最终自检时，加载 `steering/03-implementation-and-check.md`

# Default Rules

## 输出顺序

- 先结构化说明，再开发
- 不要直接根据截图自由发挥
- 不要因为代码复用或工程抽象牺牲还原度

## 推断值与冲突处理

- 缺失但又必须补齐的内容统一标记为 `【推断值】`
- 冲突信息要显式说明，不要默默选一个值直接实现
- 不要把推断值伪装成设计原值
- 如果某条规则需要依赖接口、业务代码或人工补充，也要明确写出来源

## 高风险误判点

- 多个 node-id 输入时，不要默认拼成平级页面模块
- 一个大节点下出现多个 frame / section / 状态稿时，不要默认当成同一页面里的静态区块
- 视觉上在底部、顶部、右上角、右下角的元素，不要仅凭位置直接决定 `absolute`、`fixed` 或 `sticky`
- 带 `node-id` 的链接若命中的是 section 根节点，不要直接把它当成最终页面节点
- `1440 / 1024 / 768 / 375` 这类子节点默认优先判断为断点，不要误判为多张独立页面
- `375-待我审批` 这类命名默认优先判断为“某断点下的业务状态”，不要误判为新的断点
- `弹窗 / 拨号 / ActionSheet / popup` 这类子节点默认优先判断为浮层，不要拼回主页面

## 字体策略

- 默认忽略 Figma 中的 `font-family` 原始值
- 不要在未确认授权的前提下把 Figma 字体名直接写进代码
- 优先还原字号、字重、行高、字距、对齐方式与文本容器关系

## 组件库策略

- 若使用 Arco Design，不要保留明显偏离设计稿的默认样式
- 需要显式说明输入类控件、按钮、弹窗、标签等默认样式覆盖项
- 若项目已存在全局尺寸档位、主题 token 或自定义主题包，常规组件宽度不要默认写死为 Figma 可见宽度
- 图标、图片、插图可更严格参考 Figma 尺寸；但常规组件的外层宽度与内部 icon / padding / suffix 间距必须拆开判断
- 对 `Select`、`Input`、`Button`、`Tabs`、`Tag` 等组件，优先遵守项目尺寸档位和布局基线，再做局部样式修正
- 优先级固定如下：
  1. `package.json` 中已安装 UI 框架，且当前元素可明显映射为框架组件，并且通过 `size` 即可控制主尺寸
  2. 虽未命中 UI 框架，但本地项目下存在明显可复用组件或公共样式能力
  3. 以上两种都不满足，才按手写模块处理
- 若命中前两种情况，详细文档默认只写组件映射、`size`、背景色、颜色、字号、字重、换行方式、对齐方式、阴影和必要的状态覆盖，不展开内部 `padding` / `margin`
- 除非用户明确授权“可以修改公共组件、公共样式或全局 CSS 文件”，否则默认只允许页面内适配、局部覆盖和现有组件组合，不主动改公共文件

## 详细文档策略

若用户希望直接产出可开发文档，推荐交付中必须包含：

- 数据来源与接口依赖
- 文案与计算规则
- 显隐、联动、自动回填与自动重置规则
- 资源引用清单
- 组件库覆盖清单
- 项目实现基线（含 `package.json`、公共组件目录、公共样式目录等证据来源）
- 宽度适配策略
- 框架组件映射 / `size` / 必要间距覆盖策略

# Recommended Deliverables

推荐交付顺序如下：

1. 输入材料理解结论
2. 信息来源与优先级说明
3. node-id 归类结果（若存在）
4. 页面结构树（若存在）
5. 画板归类结果（若存在）
6. 断点分组结果（若存在）
7. 本轮开发范围说明
8. 结构化 UI 设计说明书
9. 开发方案
10. 代码实现
11. 自检结论
12. 风险点、待确认项、推断值清单
